# Memory Loop 中文使用指南

> 基于 Loop Engineering 思维的项目级 AI 记忆系统

## 这是什么？

Memory Loop 是一个为 Trae IDE 打造的记忆插件。它解决了 AI agent「每次对话都从零开始」的痛点——让 agent 能够跨 session 记住项目的关键决策、踩过的坑、当前任务状态。

核心原理：**agent 写入 → 文件持久化 → harness 注入 → 下一轮 agent「看到」上次状态**。

## 核心特性

| 特性 | 说明 |
|------|------|
| 项目级记忆 | 每个项目独立维护 `.trae/memory/` 目录，互不干扰 |
| 冷热分层 | 热层（MEMORY.md）全量注入 context；冷层（archive.md）归档历史 |
| Schema 驱动 | YAML 格式的 schema 定义记忆结构，用户可自由编辑 |
| 自动压缩 | 超过 token 阈值时触发压缩，老条目归档到冷层 |
| Apple 风格前端 | 简洁的 Web UI，支持查看/编辑记忆和 schema |
| 中英双语 | 前端界面一键切换中文/英文 |
| 一键安装 | 打包成 VSIX 扩展，安装即用，无需手动启动服务器 |

## 安装方式

### 方式一：安装 VSIX 插件（推荐）

1. 获取 `memory-loop-1.0.0.vsix` 文件
2. 在 Trae 中打开扩展面板（`Ctrl+Shift+X`）
3. 点击右上角 `···` → `Install from VSIX`
4. 选择 `.vsix` 文件，安装完成
5. 重启 Trae

安装后插件会自动：
- 启动本地服务器（端口 3721）
- 在打开项目时自动初始化记忆文件
- 注册 `Ctrl+Shift+M` 快捷键打开 Dashboard

### 方式二：手动运行

如果不想安装扩展，也可以直接运行：

```bash
cd memoryloop
node server/server.js
```

然后在浏览器打开 `http://localhost:3721`。

## 快速上手

### 1. 打开 Dashboard

安装插件后，有两种方式打开记忆面板：

- **快捷键**：`Ctrl+Shift+M`
- **命令面板**：`Ctrl+Shift+P` → 输入 `Memory Loop: Open Dashboard`

### 2. 选择项目

在 Dashboard 左侧或顶栏点击「打开项目」按钮，会弹出文件夹浏览器：

- 默认显示当前用户目录
- 浏览到任意盘符根目录时，可点击「所有驱动器」切换到其他盘
- 选中目标文件夹后点击「选择此文件夹」

### 3. 自动初始化

首次打开一个项目时，系统会自动在项目下创建：

```
你的项目/
└── .trae/
    └── memory/
        ├── MEMORY.md              # 热层记忆文件（全量注入）
        ├── archive.md             # 冷层归档文件（压缩历史）
        └── memory-schema.yaml     # 记忆结构定义（可编辑）
```

### 4. 查看记忆

Dashboard 的「记忆」视图会以卡片形式展示各个 section：

- **Project Context** — 项目类型、技术栈、架构概述
- **Active Goals** — 当前正在进行的任务目标
- **Decisions Log** — 已做出的关键技术决策
- **Gotchas** — 已知的坑、约束、必须遵守的规则
- **Completed** — 最近完成的任务
- **Status** — 当前任务状态

每张卡片支持内联编辑，点击「编辑」按钮即可修改内容。

### 5. AI agent 如何写入记忆

AI agent 遵循 [SKILL.md](.trae/skills/memory-loop/SKILL.md) 协议：

- **任务开始时**：读取 MEMORY.md，了解项目上下文
- **任务完成后**：按模板槽位写入关键发现、决策、踩坑
- **超阈值时**：触发压缩，老条目归档到 archive.md

> 注意：SKILL.md 是 prompt 驱动的协议，agent 是否执行取决于是否遵循指令。在 system prompt 中强化指令效果更好。

## 前端界面说明

### 记忆视图（Memory）

- 卡片式展示各 section 内容
- 顶栏显示 token 用量进度条（绿/黄/红三色预警）
- 每张卡片可内联编辑
- 支持「压缩」按钮手动触发压缩检查

### 归档视图（Archive）

- 浏览冷层 archive.md 的压缩历史
- 支持关键词搜索 + 高亮
- 压缩后的摘要内容，不再占用热层 token

### Schema 视图（Schema）

- YAML 格式实时编辑 schema
- 可定义：section 名称、描述、最大条目数、格式模板
- 支持重置为默认 schema
- 保存后立即生效，下次写入按新 schema 执行

### 项目视图（Projects）

- 切换不同项目
- 查看最近打开的项目列表
- 文件夹浏览器支持多盘符切换

## Schema 配置详解

默认 schema 定义在 `memory-schema.yaml`：

```yaml
version: "1.0"
hot_layer:
  sections:
    - name: "Project Context"
      description: "项目类型、核心技术栈、架构概述"
      max_items: 1
      format: "prose"

    - name: "Active Goals"
      description: "当前正在进行的任务目标"
      max_items: 3
      format: "- [目标描述] | 开始时间: YYYY-MM-DD"

    - name: "Decisions Log"
      description: "已做出的关键技术决策"
      max_items: 10
      format: "- YYYY-MM-DD | 选择了X而非Y | 原因: Z"

    - name: "Gotchas"
      description: "已知的坑、约束、必须遵守的规则"
      max_items: 20
      format: "- [模块] 问题描述 → 规避方式"

    - name: "Completed"
      description: "最近完成的任务，滚动保留"
      max_items: 10
      format: "- YYYY-MM-DD | 任务描述 | 结论/产出"

compression:
  trigger_tokens: 3000        # 热层超过此 token 数触发压缩
  keep_recent_decisions: 5    # 压缩时保留最近 N 条 Decision
  keep_recent_completed: 3    # 压缩时保留最近 N 条 Completed
```

### 自定义 schema

你可以：
- 增删 section（比如加一个「API Notes」section）
- 修改 `max_items` 控制每个 section 的条目上限
- 修改 `format` 模板，让 agent 按你想要的格式写入
- 调整 `trigger_tokens` 改变压缩触发时机

> 槽位语义越窄、格式示例越具体，agent 写入的一致性越高。

## 冷热分层机制

### 热层（MEMORY.md）

- **全量注入**到 agent 的 context
- 保持精简，只放当前需要的信息
- 超过 `trigger_tokens`（默认 3000）时触发压缩

### 冷层（archive.md）

- 存储压缩后的历史摘要
- agent 可按需检索（通过 SKILL.md 协议中的检索指令）
- 不占用热层 token 预算

### 压缩规则

- 叙事类内容（决策过程、执行日志）→ 压缩后归档
- 精确事实类内容（寄存器地址、错误码、API 响应）→ 原文保留在热层
- 保留最近 N 条 Decision 和 Completed（可配置）

## 常见问题

### Q: 记忆文件是空的？

记忆需要 AI agent 按 SKILL.md 协议主动写入。如果 agent 没有遵循协议，文件只有初始化时的空占位符。可以在 Dashboard 中手动编辑，或在 system prompt 中强化写入指令。

### Q: 浏览文件夹时报错？

如果默认路径不存在，系统会自动回退到「所有驱动器」视图，从那里选择任意盘符开始浏览。

### Q: 如何切换语言？

Dashboard 顶栏有语言切换按钮（显示「中」或「EN」），点击即可切换。语言偏好会保存在浏览器 localStorage 中。

### Q: 记忆文件会污染我的代码仓库吗？

不会。所有记忆文件都在 `.trae/memory/` 目录下，这是 Trae 的管辖目录。建议在 `.gitignore` 中添加 `.trae/` 以避免提交到版本控制。

### Q: 支持哪些操作系统？

- Windows（完整支持，含多盘符浏览）
- macOS
- Linux

## 设计理念

这个项目基于 Addy 的 Loop Engineering 思想：

> "The agent forgets, the repo doesn't."
> "The memory has to be on disk and not in the context."

关键认知：**这个「记忆」不是模型的记忆，而是 Harness 的记忆**。模型本身在每次 run 都是无状态的全新实例。外部磁盘记忆是工程层对这个约束的承认和补偿。

### 为什么不做路由判断？

agent 在任务刚完成时，并不具备判断「这条信息未来是否高价值」的充分信息。所以：

- **写入层**：agent 按模板槽位填入，不做路由判断
- **冷热转换层**：定期压缩时，由 summarizer agent 决定什么归档
- **读取层**：热层全量注入，冷层按需检索

### 为什么槽位要窄？

模板槽位语义越窄，agent 写入的一致性越高。比较：

```
# 差的槽位定义
## Key Learnings
（agent 不知道该写发现、决策还是警告）

# 好的槽位定义
## Gotchas（本次发现的坑，格式：[模块] 问题描述 → 规避方式）
## Decisions（本次做出的不可逆决策，格式：选择了X而非Y，原因是Z）
```

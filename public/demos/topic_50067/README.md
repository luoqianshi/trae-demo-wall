# EmoMind（情绪之心）- 基于 BERT 中文模型的智能情绪分析系统

> 通过 TRAE 开发的端到端情绪分类 AI 系统 Demo

---

## 1. Demo 简介

### 是什么

EmoMind（情绪之心）是一个基于 **BERT-Base-Chinese + LoRA 高效微调** 的智能情绪分析系统。系统集成了 Web 前端、FastAPI 后端、LoRA 微调引擎和 13 万条中文情绪知识库，专门针对通用大模型对**中文短文本情绪识别不准**的痛点进行优化。

### 面向谁

- **内容平台**：社交媒体评论分析、实时舆情监控
- **心理咨询师**：辅助情绪评估、量化指标支持心理健康筛查
- **教育机构**：学生心理状态监测、情绪困扰早期发现
- **企业客服**：客户情绪感知、服务策略优化
- **个人用户**：情绪日记、自我认知

### 覆盖场景

舆情监控、客户服务满意度分析、心理健康初步筛查、内容审核与分级、个人情绪日记。

### 主要功能

- **情绪分类机器人**：输入一段话，AI 自动识别 6 种情绪类型（悲伤、快乐、爱、愤怒、恐惧、惊讶），支持单句和批量分析
- **LoRA 高效微调**：仅训练 0.29% 的参数，达到 99.70% 的分类准确率
- **13 万条中文知识库**：覆盖口语化、日常对话、书面表达三种风格
- **混合预测策略**：模型推理 + 关键词匹配 + DeepSeek API 三级兜底，确保 100% 响应率
- **实时训练监控**：Web 页面实时显示训练进度和评估指标
- **自学习闭环**：DeepSeek API 结果自动回写知识库，持续进化

## 2. Demo 创作思路

### 灵感来源

在日常工作和生活中，我发现中文短文本（如微博评论、聊天消息）的情绪识别是一个普遍需求，但现有方案要么依赖云端大模型 API（成本高、有隐私风险），要么对中文语境理解不足（尤其口语化表达）。这让我想到可以用轻量级的 BERT + LoRA 方案，做一个**可私有化部署、专门针对中文优化**的情绪分析工具。

### 想解决的问题

1. **通用大模型情绪识别不准**：尤其对中文短文本、口语化表达、网络用语的识别准确率不高，文化背景和语境依赖导致频繁误判
2. **缺乏中文语境优化的轻量级方案**：现有方案要么体积庞大不适合边缘部署，要么没有针对中文语境专门优化
3. **私有化部署成本高**：多数方案依赖云端 API，企业面临高昂算力和数据安全风险
4. **无法自定义情绪分类维度**：标准方案通常只提供正/负/中三分类，无法满足精细化需求

### 为什么做这个方向

选择 **BERT-Base-Chinese + LoRA** 路线的原因：

- **BERT-Base-Chinese** 是专门为中文优化的预训练模型，对中文语法和语义理解更精准
- **LoRA（Low-Rank Adaptation）** 仅需训练 0.29% 的参数即可达到全量微调效果，极大降低算力门槛，适合个人开发者和小团队
- **私有化部署**：整个模型可运行在单台服务器甚至本地机器上，数据不出域
- **6 维情绪分类**：覆盖悲伤、快乐、爱、愤怒、恐惧、惊讶，比传统三分类更贴合实际应用场景

## 3. Demo 体验地址

### 本地体验方式

下载 `emotion_mind_project.zip`，解压后进入项目目录：

```bash
cd emotion_mind_project
python3 -m http.server 8080
```

浏览器访问 `http://localhost:8080/index.html` 即可体验。

> 注意：情绪预测功能需要启动后端服务（`python backend.py`）并配置模型路径。

## 4. TRAE 实践过程

整个 Demo 从设计到可运行产品，完全在 **TRAE Work** 中完成。以下是完整开发流程：

### 4.1 需求分析与架构设计

项目起点是在 TRAE 中提出需求："做一个中文情绪分类系统"。通过与 TRAE 的多轮对话，逐步明确了技术路线：

- **模型选型**：TRAE 建议使用 `bert-base-chinese` 作为基础模型，因为它是专门为中文优化的预训练模型，对中文语法和语义理解更精准。同时采用 **LoRA（Low-Rank Adaptation）** 进行高效微调，仅需训练 0.29% 的参数即可达到全量微调效果。
- **架构设计**：确定了端到端的系统架构 —— Web 前端（HTML/CSS/JS）→ FastAPI 后端 → 混合预测引擎 → 情绪结果输出。
- **情绪分类体系**：确定 6 维情绪分类（悲伤、快乐、爱、愤怒、恐惧、惊讶），比传统正/负/中三分类更贴合实际应用场景。
- **三级兜底策略**：设计了"模型推理 → 关键词匹配 → DeepSeek API"三级预测策略，确保任何输入都有响应。

### 4.2 前端页面开发

通过 TRAE 的对话式开发，生成了完整的 6 个前端页面和 1 个统一样式表：

| 页面 | 功能 | TRAE 开发方式 |
|------|------|---------------|
| `index.html` | 统一入口首页，顶部导航栏 + iframe 框架 | 要求"做一个带导航的首页，下方用 iframe 加载其他页面" |
| `emotion_classifier.html` | 情绪分类机器人，支持单句/批量分析 | 要求"做一个情绪分析页面，输入文字显示情绪结果" |
| `knowledge_base.html` | 知识库浏览，支持筛选/搜索/分页 | 要求"做一个知识库页面，展示情绪语料数据" |
| `training_status.html` | 实时训练进度监控 | 要求"做一个训练进度页面，显示进度条和指标" |
| `lora.html` | LoRA 调优控制台，支持参数配置和评估 | 要求"做一个模型调优页面，可以配置训练参数" |
| `emotion-mind-intro.html` | 项目介绍页，展示技术亮点和架构 | 要求"做一个项目介绍页，展示产品亮点" |
| `styles.css` | Eva Design 风格统一样式表 | 要求"所有页面使用统一的 Eva Design 风格" |

**关键迭代**：
- 最初各页面有独立的导航栏，后来通过 TRAE 优化为"首页统一导航 + iframe 加载子页面"的架构，去掉了各页面的重复导航代码
- 通过 TRAE 反复调整样式细节，确保所有页面颜色、间距、圆角等视觉元素一致

### 4.3 后端服务开发

通过 TRAE 编写了完整的 FastAPI 后端服务 `backend.py`：

- **核心 API 实现**：与 TRAE 对话逐行构建，实现了 `/predict`（单条预测）、`/batch_predict`（批量预测）、`/evaluate`（模型评估）、`/train`（触发训练）、`/train/status`（训练状态查询）等接口
- **三级预测策略集成**：在 TRAE 的协助下，将 LoRA 模型推理、关键词智能匹配、DeepSeek API 兜底三种策略串联起来。当模型置信度低时，自动降级到关键词匹配；如果关键词也无法匹配，则调用 DeepSeek API 并自动将结果回写知识库
- **模型热重载**：支持在不重启服务的情况下重新加载训练好的模型，方便迭代
- **CSV 导出**：支持将批量预测结果导出为 CSV 文件

### 4.4 训练脚本开发

通过 TRAE 编写了完整的 LoRA 微调脚本 `train_lora.py`：

- **数据加载**：支持从 JSONL 文件、SQLite 知识库或内置数据加载。TRAE 帮助实现了自动检测数据来源的逻辑
- **LoRA 配置**：在 TRAE 指导下配置了 `LoraConfig`，设置 rank=8、alpha=16、target_modules=["query", "value"] 等参数
- **训练参数**：batch_size=8、learning_rate=2e-4、epochs=2，并配置了早停和学习率调度
- **Checkpoint 恢复**：实现了自动检测并恢复最近 checkpoint 的功能，训练中断后可以从断点继续
- **评估指标**：自动计算准确率、F1 分数、精确率、召回率，并保存到 `trainer_state.json`

**训练过程**：
- 第 1 个 epoch（13,170 步）：评估准确率 99.45%
- 训练过程中曾意外中断，通过 checkpoint 恢复机制成功续训
- 第 2 个 epoch 完成后：最终评估准确率 **99.70%**，F1 分数 **99.70%**，损失 **0.016**

### 4.5 数据处理与知识库构建

通过 TRAE 完成了 131,693 条中文情绪语料的构建和存储：

- **数据来源**：整合了多源中文情绪语料，覆盖口语化表达（如"好耶""美滋滋"）、日常对话（如"今天天气真好"）、书面表达（如"痛彻心扉"）三种风格
- **数据存储**：使用 SQLite 数据库 `emotion_kb.db` 存储，每条记录包含 text、label、category、type、source 字段
- **数据清洗**：通过 TRAE 编写脚本，去除重复数据、处理缺失值、统一编码格式
- **JSON 导出**：同时维护了 `emotion_kb.json` 文件，方便前端直接读取展示

### 4.6 训练监控与调试

通过 TRAE 实现了完整的训练监控体系：

- **日志监控**：编写了 `monitor_train.py` 脚本，每 2 秒读取 `train_log.txt`，解析训练进度并写入 `train_status.json`
- **前端实时展示**：`training_status.html` 每 3 秒通过 `fetch` 拉取 `train_status.json`，实时更新进度条、速度、剩余时间
- **问题排查**：训练过程中遇到进程丢失问题，通过 TRAE 分析日志，发现是内存占用过高导致系统回收进程，最终通过恢复 checkpoint 解决

0. 先和大家打个招呼

大家好，我是一名在校大学生。

说实话，在用 TRAE 之前，我脑子里一直有个想法：每次和同学约着出去玩，从找搭子、定路线、算费用到最后 AA 分钱，整个过程散落在五六个微信群和备忘录里，乱成一锅粥。我一直想做一个小工具把这件事理顺，但我的前端水平停留在"能跟着教程写个静态页面"，真正动手做一个有地图、有实时数据、有交互逻辑的完整产品，想都不敢想。

TRAE 改变了这件事。我的开发方式就是"把脑子里的想法一句句讲给它听"。比如我说"我要一个像微信小程序那样的底部导航栏，五个标签，切换的时候有动画"，它就直接给我写好了完整的 Vue 组件和 CSS。我说"地图切到地图 Tab 之后是灰屏，因为容器还没显示就初始化了"，它自己定位到生命周期时序问题，加了 nextTick 和 invalidateSize 延迟调用，一次修好。

最让我觉得"原来这么简单"的一步是地图。我原本以为集成 Leaflet 地图、调 OSRM 路线 API、在地图上画起点终点和路线这种事，没有后端根本搞不定。结果 TRAE 帮我把整个地图模块封装成一个独立的 MapController，瓦片加载失败还有三层备用源自动切换，纯前端就跑通了。这个坎跨过去之后，后面的预算计算、时间线生成、分享卡片，都是顺水推舟。

整个 Demo 从零到现在，我没有写过一行自己手敲的代码——全部是通过和 TRAE 对话完成的。它不是帮我"补全代码"，而是真的在"理解需求 → 设计方案 → 写代码 → 调试验证"这条链路上和我配合。

1. Demo 简介

是什么：同路 AA 是一个移动端优先的 Web 应用，界面模仿微信小程序风格，手机浏览器打开即可使用，也可添加到主屏幕以 PWA 模式运行。

面向谁：大学生群体，尤其是有周边游/自驾游需求、需要结伴出行和 AA 分账的学生。

主要功能：

  • AI 路线规划 + 实时数据：输入出发地和目的地，自动调用 OSRM 获取真实路线距离和车程，Open-Meteo 获取目的地实时天气，一键生成包含路线、天气、预算、风险预警、游玩攻略的完整出行方案。
  • AA 分账 + 支出可视化：自动计算油费、过路费、门票、餐饮等各项开支的人均金额，Canvas 绘制支出饼图，支持实时记账和 AA 进度追踪。
  • 交互式地图 + 沿途标注：Leaflet 地图展示起点、终点、路线 polyline，沿途标注加油站、服务区餐厅、观景台等 POI，瓦片加载失败自动切换备用源。

此外还有搭子匹配、行程时间线（根据天数动态生成多日行程）、碳足迹对比、旅行人格测试、微信群分享卡片等功能。

2. Demo 创作思路

灵感来源

这个想法来自我自己的真实经历。上学期我们四个同学约着去吐鲁沟国家森林公园玩，出发前在群里讨论了三天还没定下来：谁开车、走哪条路、油费怎么算、门票多少钱、天气怎么样要不要带外套。最后一个人垫了所有钱，回来后在群里发了七八个红包才勉强算清。我当时就想，要是有个工具能把这些事一次搞定就好了。

想解决的问题

大学生周边游有三个真实痛点：

  • 找搭子难：朋友圈发一条"有人去XX吗"，响应寥寥，缺少基于目的地和时间的结构化匹配。
  • 费用算不清：油费、过路费、门票、餐饮、停车费，项目多且分散，最后 AA 时总有人觉得不公平。
  • 路线靠百度：手动搜路线、查天气、看攻略，信息碎片化，没有一体化的出行方案。

为什么做这个方向

选择这个方向有两个原因。第一，痛点足够真实——这不是我臆想的需求，而是身边每个有车或有朋友有车的同学都遇到过的问题。第二，技术可行性高——OSRM、Open-Meteo、Leaflet 都有免费的公开 API，纯前端就能跑通核心链路，不需要后端服务器，适合作为参赛 Demo 快速验证产品想法。

在功能取舍上，我刻意控制了范围：搭子匹配、实时位置共享、SOS 这些功能目前用本地模拟数据实现并明确标注"演示模式"，不假装是真的。核心的路线规划、天气获取、预算计算、地图渲染则是真实联网的，确保评审体验时感受到的是真实可用的产品，而不是一堆假数据。

3. Demo 体验地址

本项目以 ZIP 格式打包上传，下载解压后用浏览器打开 index.html 即可体验。

4. TRAE 实践过程

开发流程

整个 Demo 的开发分为以下几个阶段，每个阶段都是通过和 TRAE 对话完成的：

  • 阶段一 - 基础架构搭建：用 Vue 3 + Pinia + CDN 搭建项目骨架，设计 5 Tab 小程序式信息架构（规划/地图/搭子/账本/我的），完成表单输入和基础 UI。
  • 阶段二 - 核心功能开发：接入 OSRM 路线 API、Open-Meteo 天气 API、Leaflet 地图，实现路线规划、天气获取、地图渲染、预算计算、AA 分账等核心链路。
  • 阶段三 - 体验打磨：添加一键演示模式、行程总览卡、动态多日时间线、分享卡片、骨架屏加载、Tab 切换动画、暗色模式等 UI/UX 优化。
  • 阶段四 - 功能完善与创新：添加 Canvas 支出饼图、碳足迹对比、旅行人格测试、沿途 POI 标注、PWA 支持、无障碍 ARIA 标签等。
  • 阶段五 - 问题修复：修复地图灰屏问题（隐藏容器初始化时序）、时间线不随天数变化、备用瓦片源自动切换等关键 Bug。

技术栈

  • Vue 3.4 + Pinia 2.1（CDN 引入，无需构建工具）
  • Leaflet 1.9 - 交互式地图
  • OSRM - 路线规划 API
  • Open-Meteo - 天气数据 API
  • Open-Meteo Geocoding - 地理编码
  • html2canvas - 分享卡片截图
  • QRCode.js - 二维码生成

总结

这个 Demo 从一个"和同学出去玩算不清钱"的小烦恼开始，最终变成了一个有路线规划、实时天气、地图导航、预算计算、搭子匹配、分享功能的完整产品。在这个过程中，TRAE 承担的不只是"写代码"的角色，更像是产品和技术之间的翻译官——我把需求用自然语言描述出来，它负责把需求变成可运行的代码。

当然，这个 Demo 还有很多可以继续完善的地方：搭子匹配目前是本地模拟数据，未来可以接入微信登录实现真实的多人同步；SOS 和实时位置共享需要后端支持才能真正可用；行程导出 PDF、行程回忆录等功能也已经有了雏形但还没完全实现。这些都是后续迭代的方向。

如果你也是大学生，也遇到过"出去玩 AA 算不清"的烦恼，欢迎下载体验，给我反馈。谢谢大家！

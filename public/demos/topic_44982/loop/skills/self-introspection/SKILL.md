# 自我反思技能（Self-Introspection Skill）

构建任务执行过程中的自我评估机制，实现错误识别、根因分析、经验沉淀与持续改进。

## 与 loop engineering 联动

loop engineering 每轮迭代的 [H] 本轮根因反思 和 [Check] 经验沉淀 两个步骤必须调用本 skill：
- **[H] 每轮结束**：用五问法（5-Whys）分析本轮每个修复的根因，输出"现象→为什么→根因→预防"
- **[Check] 经验沉淀**：提炼新 Bug 模式，更新到 project_memory.md 的"Bug 模式库"

**反例**：loop engineering 6 轮只用了 1 个 skill，没在 [H] 调用本 skill，导致流程缺陷被遗漏。

---

## 一、自我反思触发时机

| 时机 | 触发条件 | 反思重点 |
|------|----------|----------|
| 任务完成时 | 一个功能/修复/Bug 处理完毕 | 输出是否正确？有无遗漏？ |
| 遇到错误时 | 代码报错/测试失败/用户反馈 | 根因是什么？如何避免复现？ |
| 多轮迭代时 | loop engineering 某轮结束 | 本轮发现什么模式？下轮聚焦什么？ |
| 关键决策后 | 做了架构/算法选型 | 选型是否合理？有无更好方案？ |
| 对话结束前 | 用户任务即将完成 | 全流程有无遗漏？经验是否记录？ |

## 二、错误识别机制

### 2.1 四层错误检测

```
Layer 1: 语法层 → 代码能否运行？
  → 检查：括号匹配、分号、语法结构
  → 工具：浏览器控制台报错

Layer 2: 逻辑层 → 运行结果是否正确？
  → 检查：边界值、条件分支、状态转换
  → 工具：验证锚点（VDOT=50 → E:4:58）

Layer 3: 交互层 → 用户体验是否正常？
  → 检查：点击响应、输入失焦、事件冒泡
  → 工具：浏览器实测 + 截图

Layer 4: 隐患层 → 是否有潜在风险？
  → 检查：全局状态共享、CSS 变量引用、内存泄漏
  → 工具：Grep 搜索坏味道模式
```

### 2.2 错误分类与处理策略

| 错误类别 | 示例 | 处理策略 |
|----------|------|----------|
| 事实性错误 | 公式参数写反 | 立即修正 + 记录 |
| 逻辑性错误 | 边界条件遗漏 | 修正 + 添加边界测试 |
| 一致性错误 | CSS 变量未定义 | 修正 + 统一变量表 |
| 性能问题 | 整页 reRender | 优化为局部更新 |
| 设计缺陷 | 全局状态共享 | 重构为实体属性 |

## 三、根因分析方法

### 3.1 五问法（5-Whys）

```
现象：心率区间显示 85-105 bpm，严重偏低
  ↓ 为什么？
因为计算公式是 restingHr + 30~50
  ↓ 为什么用这个公式？
因为没有对照运动生理学标准，随手写的偏移量
  ↓ 为什么没发现？
因为只做了静态分析，没验证数值合理性
  ↓ 为什么没验证？
缺乏"验证锚点"机制
  ↓ 根因：缺少领域知识验证环节

修复：改用 Karvonen 公式 + 添加验证锚点
预防：所有公式必须标注来源 + 验证锚点
```

### 3.2 鱼骨图分析

```
问题：VDOT 计算产生不现实配速（1:39/km）

人（Human）    → 公式理解不深入
法（Method）   → 用了简化拟合而非官方表
料（Material） → 没有官方配速表数据
环（Environment）→ 缺少验证标准
测（Measure）  → 没有验证锚点

根因：方法+测量 缺失
修复：换用官方配速表插值 + 添加验证锚点
```

### 3.3 时序回溯法

```
Bug：滑块值不持久

时间线：
1. 首次渲染 → 显示滑块（无值）
2. 用户拖动 → oninput 读取 value
3. 点击保存 → saveReadiness() 读取 value → 存入 STORE
4. render() → 重新渲染 → 滑块重置为默认值

断裂点：步骤4，render() 重建了滑块，未回显旧值
根因：编辑态与显示态未分离
修复：添加 editingReadiness 标志，编辑时不覆盖旧值
```

## 四、经验沉淀机制

### 4.1 Bug 模式库

每次修复 Bug 后，提炼为可复用的模式：

```markdown
### Bug 模式：[模式名]
- **症状**：____（用户可观察到的现象）
- **根因**：____（代码层面的原因）
- **修复**：____（具体改了什么）
- **预防**：____（如何避免再次出现）
- **检查方法**：____（如何快速发现）
- **关联文件**：____
```

### 4.2 已沉淀的 Bug 模式

| 模式 | 症状 | 预防方法 |
|------|------|----------|
| 全局状态共享 | 不同实体数据互相覆盖 | 状态应绑定到实体属性 |
| CSS 变量无效 | 样式回退异常 | Grep `var(--` 对比 `:root` |
| 事件冒泡 | 按钮触发父级事件 | 添加 stopPropagation |
| 时区偏移 | getDay() 返回非预期 | setHours(0,0,0,0) 归零 |
| IME 失焦 | 中文输入被打断 | onchange 替代 oninput |
| 公式参数错 | 计算偏离常识 | 标注来源 + 验证锚点 |

### 4.3 经验记录流程

```
修复 Bug
  → 提炼模式（是否见过类似 Bug？）
  → 记录到模式库
  → 更新检查清单
  → 下轮迭代时用 Grep 扫描同类问题
```

## 五、持续改进流程

### 5.1 PDCA 循环

```
Plan（计划）
  → 确定本轮迭代维度
  → 制定检查清单
  → 设定验证标准

Do（执行）
  → 按检查清单逐项扫描
  → 发现问题立即修复
  → 记录修复过程

Check（检查）
  → 浏览器实测验证
  → 控制台错误检查
  → 对比修复前后效果

Act（改进）
  → 提炼 Bug 模式
  → 更新检查清单
  → 调整下轮维度优先级
```

### 5.2 自我评估清单

每轮迭代结束后，回答以下问题：

```
□ 本轮发现了多少个问题？
□ 其中 P0/P1 级别多少个？
□ 修复后是否引入了新问题？
□ 是否有遗漏的检查项？
□ 是否发现了新的 Bug 模式？
□ 这个模式是否可能在其他模块也存在？
□ 检查清单是否需要更新？
□ 下一轮应该聚焦什么维度？
```

### 5.3 经验复用

```
新 Bug 出现
  → 查 Bug 模式库 → 有类似模式？
    → 是：直接套用修复方案 + 验证
    → 否：用五问法分析根因 → 沉淀新模式
  → 全局扫描同类问题（Grep 搜索）
  → 一次性修复所有同类 Bug
```

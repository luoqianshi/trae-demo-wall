# TRAE 创造力大赛监控 — 本地 Web 版需求文档

## 1. 项目概述

为 TRAE AI 创造力大赛（https://forum.trae.cn）构建一个本地运行的数据监控网页。

- **核心目标**：用网页替代现有 CLI 工具，直观展示 700+ 条参赛帖子的数据，支持排序、搜索、查重、趋势分析。
- **技术路线**：Flask 后端 + 原生 HTML/JS/CSS 前端，零构建工具，双击启动即用。
- **运行方式**：`start.bat` 启动 Flask 服务，浏览器自动打开。服务保持运行，每晚 20:00 自动更新数据，也可手动触发增量更新。
- **赛事周期**：2026-06-16 至 2026-08-22（决赛结束）。到期后自动更新暂停。

---

## 2. 技术架构

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端框架 | Flask | 单进程运行，REST API 供前端调用 |
| 定时任务 | APScheduler | 后台线程，每天 20:00 触发更新 |
| 前端 | 原生 HTML + JS + CSS | 单页应用（SPA），5 个 Tab 切换，无 npm 构建 |
| 数据存储 | JSON 文件 | `data/posts.json` 主数据 + `data/snapshots/` 历史快照 |
| 图表 | 原生 Canvas | 趋势折线图，不引入外部图表库 |
| 启动脚本 | `start.bat` | Windows 双击启动，自动打开浏览器 |

## 3. 数据模型

### 3.1 帖子记录（Post）

每条帖子为一个 JSON 对象，字段与现有 `scraper.py` 保持一致：

```json
{
  "id": 12345,
  "title": "帖子标题",
  "excerpt": "摘要前200字",
  "created_at": "2026-06-16T08:30:00.000Z",
  "updated_at": "2026-06-20T10:00:00.000Z",
  "views": 1500,
  "like_count": 89,
  "posts_count": 12,
  "reply_count": 11,
  "tags": ["学习工作"],
  "author": "username",
  "url": "https://forum.trae.cn/t/12345",
  "content": "纯文本正文内容...",
  "raw_html": "<p>原始HTML正文...</p>"
}
```

**新增字段**：
- `raw_html`：原始 HTML 正文（用于抽屉详情渲染，不用于搜索）

### 3.2 历史快照（Snapshot）

每天第一次成功更新后，将完整 `posts.json` 复制一份到：

```
data/snapshots/2026-06-22.json
data/snapshots/2026-06-23.json
...
```

**保留策略**：最多保留 30 份快照，超出自动删除最早的。由 `snapshot.py` 管理。

### 3.3 更新状态（UpdateStatus）

更新任务运行时，实时写入 `data/status.json`：

```json
{
  "state": "running",
  "started_at": "2026-06-22T20:00:00",
  "message": "正在更新第 123 / 700 条帖子",
  "current": 123,
  "total": 700,
  "new_posts": 5
}
```

状态枚举：`idle`（空闲）、`running`（运行中）、`success`（成功）、`error`（失败）。

## 4. 核心修复：数据更新策略

### 4.1 问题背景

现有 `scrape_all(incremental=True)` 的逻辑是"遇到已有 ID 就跳过"，这导致：
- 旧帖子的点赞、浏览、评论数永远不会更新
- 排行榜按点赞/浏览排序的结果越来越不准
- 趋势图没有真实的历史变化数据

### 4.2 修复后的更新流程

增量更新改为**两阶段执行**：

```
┌─────────────────────────────────────────────────────────┐
│  阶段一：刷新已有帖子数据（Update Existing）              │
│  - 加载 data/posts.json 中所有已有帖子 ID                │
│  - 逐个重新请求 /t/{id}.json 获取最新详情                │
│  - 更新 views / like_count / posts_count / reply_count   │
│  - 更新 updated_at / content / raw_html                  │
│  - 每条间隔 1-2 秒，显示进度（第 X / 总 Y 条）           │
├─────────────────────────────────────────────────────────┤
│  阶段二：增量爬取新帖（Fetch New）                        │
│  - 从第 1 页开始翻页获取列表                             │
│  - 遇到已有 ID 则跳过，直到连续整页都是已有 ID 时停止      │
│  - 新帖获取详情后追加到数据文件                            │
├─────────────────────────────────────────────────────────┤
│  阶段三：保存与快照（Save & Snapshot）                    │
│  - 保存更新后的 posts.json                               │
│  - 如果是当天第一次成功更新，保存一份快照到 snapshots/     │
│  - 清理超过 30 天的旧快照                                │
│  - 将状态置为 success / error                            │
└─────────────────────────────────────────────────────────┘
```

**性能估算**：700 条帖子 × 1.5 秒平均间隔 ≈ 17 分钟。对于每晚自动更新 + 手动更新来说可以接受。界面需要显示实时进度。

### 4.3 并发控制

任何时刻只能有一个更新任务在运行。

- **实现方式**：Flask 单进程模式下，用一个全局变量 `is_updating = False` 作为内存锁。
- **触发时机**：
  - 定时任务到达时，如果发现锁为 `True`，跳过本次执行，并在日志中记录"跳过：上次更新尚未完成"。
  - 用户点击手动更新时，如果锁为 `True`，前端弹出提示"正在更新中，请稍后再试"，不触发新请求。

### 4.4 赛事结束处理

- 在 `config.py` 中定义 `EVENT_END_DATE = "2026-08-22"`。
- 每次启动时检查：如果当前日期 > EVENT_END_DATE，自动更新任务不再执行，界面顶部显示横幅提示"赛事已结束，自动更新已暂停。如需继续监控，请手动点击更新。"
- 连续 7 天检测到无新增帖子时，也在日志中提示"近期无新增数据，建议关闭自动更新"。

## 5. 后端 API 设计

### 5.1 API 列表

| 方法 | 路由 | 说明 |
|------|------|------|
| GET | `/api/posts` | 获取帖子列表（支持排序、分页、过滤） |
| GET | `/api/posts/<id>` | 获取单条帖子详情 |
| GET | `/api/search?q=xxx&demo_only=1` | 关键词搜索 |
| GET | `/api/similar?topic_id=xxx&threshold=0.5&demo_only=1` | 相似作品搜索（基于指定帖子反查） |
| GET | `/api/tracks` | 赛道分析数据（含各赛道关键词分布和帖子列表，结果缓存） |
| GET | `/api/trend?topic_id=xxx&days=14` | 获取帖子或全站趋势数据 |
| GET | `/api/status` | 获取当前更新状态（用于轮询） |
| POST | `/api/update` | 手动触发增量更新（需密码） |
| GET | `/api/export?format=csv` | 导出数据（CSV 或 JSON） |

### 5.2 关键接口详情

**GET /api/posts**

查询参数：
- `sort`：排序字段（`views` / `likes` / `comments` / `created`），默认 `views`
- `order`：排序方向（`asc` / `desc`），默认 `desc`
- `page`：页码，默认 1
- `per_page`：每页条数（20 / 50 / 100），默认 50
- `demo_only`：是否仅返回参赛帖（0 或 1），默认 1
- `track`：按赛道过滤（可选）

返回：
```json
{
  "posts": [...],
  "total": 707,
  "page": 1,
  "per_page": 50,
  "total_pages": 15
}
```

**GET /api/trend**

查询参数：
- `topic_id`：指定帖子 ID（可选，不传则返回全站趋势）
- `days`：返回最近 N 天数据（7 / 14 / 30），默认 14

返回：
```json
{
  "dates": ["06-09", "06-10", "06-11", ...],
  "views": [1200, 1250, 1300, ...],
  "likes": [80, 85, 89, ...],
  "rank": [15, 14, 12, ...]
}
```

**POST /api/update**

请求体：
```json
{"password": "your_password"}
```

响应：
- 密码正确且当前未在更新 → `{"success": true, "message": "更新任务已启动"}`
- 密码错误 → `{"success": false, "message": "密码错误"}`
- 正在更新中 → `{"success": false, "message": "正在更新中，请稍后再试"}`

## 6. 前端页面设计

### 6.1 整体布局

```
┌─────────────────────────────────────────────────────────────┐
│  <header>  TRAE 创造力大赛监控  │  最后更新：2026-06-22 20:05   │
├─────────────────────────────────────────────────────────────┤
│  排行榜  │  赛道分析  │  数据管理                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    <Tab 内容区域>                            │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  <footer>  TRAE AI 创造力大赛  │  数据来源于 forum.trae.cn     │
└─────────────────────────────────────────────────────────────┘
```

- **顶部导航**：3 个 Tab 标签，点击切换内容区。
- **URL 持久化**：当前 Tab、搜索关键词、排序字段写入浏览器 URL（如 `/#tab=leaderboard&sort=likes&order=desc`），刷新页面后自动恢复状态。
- **赛事结束横幅**：如果当前日期 > EVENT_END_DATE，顶部显示黄色横幅提示。

### 6.2 暗色主题规范

基于 Trae 官网风格：

| 元素 | 值 |
|------|-----|
| 页面背景 | `#0a0a0f` |
| 卡片/面板背景 | `#111118`（边框 `rgba(255,255,255,0.06)`） |
| 主文字 | `#e0e0e6` |
| 次要文字 | `#8a8a96` |
| 强调色 | 蓝紫渐变 `#4d7cfe` → `#a855f7` |
| 表格行 hover | `rgba(255,255,255,0.03)` |
| 抽屉遮罩 | `rgba(0,0,0,0.6)` |
| 赛道标签色 | 生活娱乐`#22c55e` / 学习工作`#3b82f6` / 社会服务`#f97316` / 硬件交互`#a855f7` / 社会公益`#ec4899` |

### 6.3 Tab 1: 排行榜（默认首页）

**核心表格**：
- 列：`排名 | 标题 | 作者 | 赛道标签 | 点赞 | 浏览 | 评论 | 发布时间 | 操作`
- **操作列**：每行末尾显示「相似」按钮，点击后打开右侧「相似作品」抽屉（见 6.7），以该帖子为锚点展示相似作品列表，无需跳转到其他 Tab
- **排序交互**：点击表头切换升降序，列头旁显示 ▲ / ▼ 箭头。
- **分页**：底部传统分页栏，可选每页 20 / 50 / 100 条。
- **标题截断**：标题列固定最大宽度，超出部分显示 `...`，鼠标悬停时 tooltip 显示完整标题。
- **点击行**：右侧滑出抽屉面板（见 6.6）。

**首次无数据状态**：
- 如果 `posts.json` 不存在，表格区域显示："暂无数据，点击下方按钮开始首次全量爬取" + 「开始爬取」按钮。

### 6.4 Tab 2: 赛道分析

改为**单赛道看板**，参考黑客松作品分析平台（Devpost Gallery / Hack-Hackathon）的做法：不做自动分类命名，展示可量化数据让用户自行探索。

**赛道切换器**（看板左上角）：
- 5 个 Tab：生活娱乐 / 学习工作 / 社会服务 / 硬件交互 / 社会公益
- 默认选中第一个赛道，切换后看板内容刷新
- 旁侧显示未分类帖子总数

**看板主区域**（从上到下）：

1. **赛道概览栏**
   - 帖子数 / 总参赛帖数 / 占比百分比
   - 拥挤状态标签：占比 > 40% 红色「拥挤」/ > 25% 黄色「关注」/ 其他 绿色「正常」

2. **关键词分布区**（替代聚类分类）
   - 提取该赛道所有帖子标题中有意义的关键词（jieba 分词 + 停用词过滤，仅限标题不含正文）
   - 按出现频率降序展示前 15 个关键词，柱状图呈现
   - **点击关键词** → 下方帖子列表筛选只显示标题含该关键词的帖子
   - 可多选关键词（交集筛选），再次点击已选关键词取消
   - 未选中任何关键词时显示全部帖子

3. **帖子列表**
   - 默认显示该赛道浏览量前 50 帖子，支持浏览/点赞/评论排序
   - 每行显示：标题 / 作者 / 浏览 / 点赞 / 评论
   - 点击行打开详情抽屉

**设计理念**：
- 不替用户判断"这类叫什么"，而是把数据透明呈现，让选手自己探索赛道内大家都在做什么
- 关键词可交互筛选，选手关心的主题点一下就能看相关帖子

### 6.5 Tab 3: 数据管理

分为四个区域：

| 区域 | 内容 |
|------|------|
| **手动更新** | 按钮「增量更新」→ 弹出密码输入框 → 验证通过后启动更新 → 按钮变为「更新中...」不可点击 |
| **更新进度** | 显示当前更新状态：空闲 / 进度条（第 X / Y 条，新增 Z 条）/ 成功 / 失败 |
| **更新日志** | 最近 10 次更新记录，格式：时间 / 耗时 / 新增条数 / 状态 |
| **趋势图表** | 全站帖子数 / 总点赞 / 总浏览 的 7/14/30 天趋势折线图（Canvas） |
| **数据导出** | 「导出 CSV」和「导出 JSON」按钮，点击直接下载 |

**更新失败提示**：
- 如果更新失败，进度区域显示红色错误信息，如"更新失败：网络连接超时，请检查网络后重试"。

### 6.6 抽屉详情面板

右侧滑出，宽度占屏幕 40%，半透明黑色遮罩（点击遮罩关闭）。

内容：
- 标题（大字号）
- 作者 + 发布时间
- 赛道标签（彩色 pill）
- 数据卡片：点赞 / 浏览 / 评论（横向排列）
- **趋势小图**：该帖子的点赞 & 浏览量 7 天变化折线图（基于快照数据）
- 正文内容（渲染 `raw_html`，需过滤危险标签）
- 「在论坛中查看」按钮（外链打开原帖）

**正文安全过滤**：
- 只允许以下标签：`p, br, strong, em, a, img, ul, ol, li, blockquote, h1, h2, h3, h4, h5, h6`
- 移除所有 `script, style, iframe, form, input, button` 等标签
- 所有 `a` 标签加 `rel="noopener noreferrer"` 和 `target="_blank"`
- `img` 标签只保留 `src` 和 `alt` 属性

### 6.7 相似作品抽屉

从排行榜表格点击「相似」按钮时，右侧滑出**独立抽屉**，展示该帖子的相似作品。与详情抽屉（6.6）是两套独立的抽屉，可分别打开和关闭。

**触发方式**：排行榜/搜索表格每行的「相似」按钮。

**抽屉内容**：

- **头部**：标题「相似作品」+ 关闭按钮
- **锚点帖子摘要区**：标题 / 作者 / 赛道标签 / 摘要前 100 字
- **相似作品列表**：每条显示
  - 相似度百分比（≥0.85 红色 / ≥0.7 橙色 / 其他黄色）
  - 标题
  - 作者 + 赛道标签
  - 浏览量 + 点赞数
  - 摘要前 100 字
- **阈值滑块**：0.30 ~ 1.00，默认 0.50，拖动后实时刷新结果
- **仅参赛帖开关**：默认开启

**交互行为**：

- 点击相似作品列表中的任意一行 → 打开**详情抽屉**（6.8），展示该帖子的完整信息和趋势图
- 相似抽屉和详情抽屉可同时打开（详情抽屉覆盖在相似抽屉之上）
- 点击遮罩关闭当前最上层抽屉
- 关闭相似抽屉不会影响详情抽屉

**空状态**：
- 无相似结果时，列表中显示"未找到与该帖子相似度超过阈值的其他作品"

**宽度**：屏幕的 40%，与详情抽屉一致。

## 7. 安全与边界处理

### 7.1 更新任务安全

- **并发锁**：全局内存锁确保同时只有一个更新任务。
- **异常处理**：更新过程中任何帖子请求失败，记录错误并继续处理下一条，不中断整个任务。
- **网络超时**：单帖请求超时 30 秒，重试 3 次，仍失败则跳过。

### 7.2 数据安全

- **密码验证**：手动更新密码存储在 `config.py` 的 `UPDATE_PASSWORD` 中，后端比对，前端不存储。
- **XSS 防护**：抽屉详情中的 `raw_html` 必须经过白名单过滤后再渲染。

### 7.3 边界场景

| 场景 | 处理方式 |
|------|----------|
| 首次启动无数据 | 提示用户开始首次全量爬取 |
| 更新中关闭浏览器 | 后端继续执行，下次打开页面轮询状态即可 |
| 赛事结束后打开 | 顶部横幅提示自动更新已暂停 |
| 快照文件损坏 | 跳过损坏的快照，继续读取其他快照 |
| 某条帖子被论坛删除 | 保留本地数据，但在详情页提示"该帖子可能已被删除" |
| 搜索无结果 | 显示友好空状态，不报错 |
| 表格只有 1 页数据 | 隐藏分页栏 |

## 8. 开发阶段

### Phase 1: 基础骨架与数据更新修复（核心优先）

1. **Flask 应用骨架**：`app.py` + `templates/index.html` + `static/css/style.css` + `static/js/app.js`
2. **修复数据更新策略**：
   - 修改 `scraper.py`，将增量更新改为"刷新已有数据 + 爬取新帖"两阶段
   - 新增 `snapshot.py`：快照保存、读取、自动清理（保留30天）
   - 新增更新状态写入 `data/status.json`
3. **数据管理 Tab**：
   - 手动更新按钮 + 密码弹窗
   - 更新进度轮询显示
   - 更新日志展示
4. **启动脚本**：`start.bat`

**验收标准**：
- `start.bat` 双击后自动打开浏览器，显示网页
- 点击「增量更新」输入正确密码后，后台开始更新，页面显示实时进度（第 X / Y 条）
- 更新完成后，`data/posts.json` 中旧帖子的 views/likes 被更新，新帖子被追加
- 每天第一次成功更新后，`data/snapshots/` 下出现当天日期的快照文件
- 超过 30 天的旧快照自动被删除
- 如果当前日期 > 2026-08-22，自动更新不执行，页面显示赛事结束横幅

### Phase 2: 排行榜与抽屉详情

1. **排行榜表格**：表头排序、分页（20/50/100）、每页条数切换
2. **赛道标签渲染**：彩色 pill 标签
3. **抽屉面板**：右侧滑出、帖子详情、数据卡片
4. **首次无数据检测**：自动提示首次爬取

## 11. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 700 条帖子全量刷新需 15~20 分钟 | 用户等待时间长 | 界面显示实时进度条 + 预计剩余时间；自动更新在后台静默执行 |
| 频繁请求可能触发论坛限流 | 爬取中断 | 保持 1~2 秒随机间隔；单帖失败重试 3 次后跳过，不中断整体任务 |
| 帖子正文 HTML 含恶意脚本 | XSS 风险 | 使用 bleach 白名单过滤，只允许安全标签 |
| start.bat 窗口被用户误关 | 服务停止，无法自动更新 | 在页面顶部显示"服务运行中"状态指示器；引导用户最小化而非关闭 |
| 赛事结束后数据不再变化 | 自动更新做无用功 | 到期自动暂停，提示用户 |

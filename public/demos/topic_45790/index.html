研复通 解决什么

代码能运行，不等于论文复现正确。研复通不是“帮你总结论文”，而是回答 6 个问题：

  1. 论文方法是否在源码中真正实现；
  2. 源码虽然能运行，但是否偏离论文流程；
  3. 实验参数、数据预处理和评价指标是否一致；
  4. 当前项目究竟复现到了哪一步；
  5. 下一步应该修改什么；
  6. 每个判断是否具有可追溯证据。
术语速查（首次出现时已附简短解释）
  • message_id：每条消息的唯一编号
  • 字段边界：一个字段在消息字节流中的起止位置
  • 严格分组：只有结构完全一致的消息才能一起送入模型
  • 可追溯：每个结论都能回到原消息、源码或实验记录
  • GRU request：提交给语义模型的一组结构化字段数据

本 Demo 已实现核心审计闭环：载入案例 → 执行审计 → 查看风险证据 → 生成修正计划 → 模拟应用修复 → 重新审计 → 导出报告。所有结论由本地规则引擎 R01–R12 计算得出，未写死任何结论。

关于本 Demo

请如实了解本初赛 Demo 的能力范围：

  1. 这是初赛可交互原型，用于演示产品定位与核心闭环。
  2. 当前版本采用本地结构化规则引擎（R01–R12），根据导入数据计算风险与可信度。
  3. 内置 MDI+GRU 案例来源于真实科研开发流程，但已脱敏，不包含任何私人源码或数据。
  4. 页面不会声称已经自动理解任意 PDF 或任意源码；如需解析论文与静态分析代码，需未来版本接入。
  5. 未来版本计划接入论文解析和代码静态分析模块。
  6. 「应用示例修复」只修改演示数据，不修改用户真实工程文件。

不得把规则演示冒充大模型自动理解任意项目。所有计算在浏览器本地完成，不上传任何文件。

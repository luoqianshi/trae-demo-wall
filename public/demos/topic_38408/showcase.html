技术创新

五个创新点，让自动记账真正可用

从通知监听到智能解析，从 ROM 保活到四层过滤，每个创新点都解决了一个真实的技术难题。

01
🔔

通知监听「后门」方案

利用 Android NotificationListenerService 合法读取支付通知，无需 root、无需对接支付平台 API。前台服务 + 低优先级通知保活，断连自动重连，启动时扫描已有通知补录漏记记录。

0需要 root / API 对接
02
🧩

14 平台智能解析引擎

支持微信、支付宝、拼多多、云闪付、美团、京东、5 大银行、抖音、快手、滴滴。5 级金额提取正则逐级降级匹配，9 级商户名提取，智能渠道识别（含拼多多「支付宝调用银行卡」复合渠道）。

14支付平台支持
03
🛡️

国产 ROM 保活体系

通过反射 SystemProperties.get 读取 ro.* 属性，精准识别 MIUI/HyperOS/ColorOS/EMUI/OriginOS/OneUI 六大 ROM。心跳检测 + 看门狗巡检 + 自动重启三重保障，确保通知监听不被系统杀死。

6ROM 精准识别
04
🧹

四层过滤防误判

黑名单 30+ 关键词拦截统计和营销通知，收入通知过滤 20+ 关键词只记支出，金额范围校验 0.01~99999.99 元，基于 notificationId 防重复去重。四层过滤确保只记录真实消费。

4层过滤机制
05
📦

Product Flavors 双版本架构

利用 Android Gradle Product Flavors 在编译期拆分 Standard/Pro 两个独立版本。Standard 零 Pro 残留代码，Pro 增加理财持仓管理、收益排行、MySQL 局域网同步。两版本包名不同可共存。

2版本编译期隔离

真实踩坑案例（摘自技术文档）

StateFlow vs LiveData 混用
ViewModel 中使用 StateFlow 但 UI 层错误调用 LiveData 专用的 observeAsState。Trae 定位后改为 collectAsStateWithLifecycle。
Hilt 与 Kotlin 2.1.x 不兼容
Kotlin 2.1.x 的 Metadata 版本 2.1.0 超出旧版 Hilt 支持上限 2.0.0。Trae 建议升级 Hilt 到 2.56+。
ViewModel 无法实例化闪退
MainViewModel 构造函数需要 Repository 参数但未接入 Hilt 注入。Trae 指导完成 ViewModel、Activity、Application 三处注解改造。
Composable 函数存入列表返回 Unit
将 HomeScreen() 直接调用后存入列表返回 Unit 导致页面空白。Trae 指出需用 lambda 包裹实现延迟调用。

# 实时统计聚合器的快照读期不一致

## 背景

`LiveStats` 是服务端实时统计器：每来一个样本调用 `add(value)`，**同时**更新三处状态——总计数 `count`、样本总和 `sum`、以及样本所属分桶 `bucket(value % NB)`。它对外提供 `snapshot()` 拍一份当前统计视图，**该方法允许在其它线程正在 `add()` 的过程中被并发调用**（用于监控/上报）。

它对外保证：**任意一份 `snapshot()` 都必须自洽**——

1. `count == Σ buckets`（计数等于各分桶之和）。
2. `sum` 与已计入的样本一致（本题样本值恒为 1，故 `sum == count`）。
3. 全部 `add()` 结束后，`count` 精确等于实际调用次数，无丢失。

## 现象

- 单线程、低并发完全正常：快照怎么读都自洽。
- 高并发下，**在写入进行中拍的快照偶发对不上**：`count` 与各桶之和不相等，且每次跑出的差值还不一样。所有 `add()` 结束后再读则又恢复正常。

## 你的任务

定位并修复缺陷，使得**任意并发程度下、任意时刻**拍出的 `snapshot()` 都严格满足上述三条不变量。

## 约束

- **只能修改 `buggy/solution.cpp`**，不得改动测试或调用方式。
- 保持 `add(long)`、`count()`、`sum()`、`bucket(int)`、`snapshot()`、`run_concurrent(int,int)`、`Snap` 结构、静态成员 `NB` 的对外签名与语义不变。
- 注意：仅把各字段替换为原子类型并不足够——`snapshot()` 逐字段读取的过程本身也会与并发 `add()` 交错，读到跨字段的中间态；只给 `add()` 或只给 `snapshot()` 单边加锁同样无法消除这种读期撕裂。

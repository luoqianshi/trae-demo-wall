## 一、背景与问题

PRD 原始产品灵魂链路：
```
告诉 AI 你的城市/心情/预算 → AI 生成密令地图 → 出发闯关
```

主应用集成阶段 1 把 home 简化成了"三选一主题卡菜单"，丢失了"输入意图 → AI 理解 → 生成"这个交互闭环。打字机和加载动画只是包装，没有"输入"环节，AI 沦为播报员而非指挥官。PRD 假设表里预判的"评委可能觉得不是真AI"风险被兑现。

## 二、目标

在不联网、`file://` 单文件约束下，通过恢复"输入闭环"让 AI 生成感立住：
1. **输入闭环** — 用户用 chip 描述场景/预算/人数，不是选卡
2. **个性化回应** — AI 打字回应时引用用户的实际选择，营造"它听懂了"
3. **生成过程可视化** — AI"思考"分步动画（分析需求→匹配地点→规划路线→生成密令）

## 三、流程变更

### 现状流程
```
boot → home(三选一选卡) → dialog(打字briefing) → 引擎 → result → share
```

### 新流程
```
boot → ai-chat(chip输入意图) → ai-generating(AI思考动画) → dialog(briefing) → 引擎 → result → share
```

新增两个 DOM 页面 `ai-chat`、`ai-generating`，替换原 `home` 的选卡逻辑。`dialog`（briefing 打字）保留不变。

## 四、页面设计

### 4.1 ai-chat（AI 对话输入页，替代 home 选卡）

**布局（自上而下）：**
```
┌─────────────────────────────────┐
│  [AI头像]  AI 指挥官             │  ← 像素头像 + 称谓
│  "特工，密令系统已激活。          │  ← AI 开场白（打字效果）
│   告诉我你的周末计划。"          │
├─────────────────────────────────┤
│  场景                            │  ← chip 分组
│  [情侣约会✓] [独行漫游🔒] [好友聚会🔒] │
│                                  │
│  预算                            │
│  [¥100] [¥200✓] [¥300]          │
│                                  │
│  人数                            │
│  [1人] [2人✓] [4人]             │
├─────────────────────────────────┤
│       [ 生成密令 ▶ ]             │  ← 选完场景才可点
└─────────────────────────────────┘
```

**交互：**
- AI 开场白用打字机效果逐字显示（0.8s 内完成，快）
- chip 三组：场景（必选）/ 预算（默认¥200）/ 人数（默认2人）
- 场景 chip：情侣约会可选，其余两个显示🔒即将开放（阶段 2 数据就绪后开放）
- 预算/人数 chip：可自由切换，不影响实际关卡数据，但会被 AI 回应引用
- "生成密令"按钮：选中场景后高亮可点

### 4.2 ai-generating（AI 生成页，替代 dialog 前半）

**布局：**
```
┌─────────────────────────────────┐
│  [AI头像]  AI 指挥官             │
│  "收到。情侣约会，预算¥200，     │  ← 动态拼接用户选择
│   2人出行。正在为你规划          │
│   深圳南山专属路线..."           │
├─────────────────────────────────┤
│  ▸ 分析需求          ✓          │  ← 四步思考动画
│  ▸ 匹配深圳南山坐标  ✓          │     依次亮起
│  ▸ 规划4关路线       ✓          │     每步约0.6s
│  ▸ 密令生成完毕!     ✓          │
├─────────────────────────────────┤
│        [ 进入任务 ▶ ]            │  ← 全部完成后可点/自动进
└─────────────────────────────────┘
```

**交互：**
- AI 回应文案动态拼接：`"收到。${场景}，预算¥${预算}，${人数}人出行。正在为你规划深圳南山专属路线..."`
- 四步思考动画依次亮起，每步约 0.6s，共约 2.4s
- 全部完成后自动进入 dialog（briefing 打字），或显示"进入任务"按钮

## 五、数据扩展

在 `WM.THEMES['shenzhen-couple']` 增加场景元数据（供 chip 渲染与 AI 回应引用）：

```js
WM.THEMES['shenzhen-couple'] = {
  // ...现有字段...
  scene: {
    label: '情侣约会',
    city: '深圳南山',
  },
}
```

chip 的可选项硬编码在 app.js（场景/预算/人数三组），不进主题数据。预算/人数只影响 AI 回应文案，不改变 stages。

## 六、文件改动

| 文件 | 改动 |
|------|------|
| `index.html` | 新增 `#page-ai-chat`、`#page-ai-generating` 两个 DOM 页面结构 + CSS；移除 `#page-home` 的选卡结构（改为 ai-chat） |
| `js/app.js` | 新增 `renderAiChat` / `startAiGenerating` 逻辑；chip 选择状态管理；AI 回应动态拼接；四步思考动画；`pages` 数组增加两项 |
| `js/stages.js` | `shenzhen-couple` 主题增加 `scene` 字段（由 generate-stages.js 重新生成） |
| `generate-stages.js` | 输出 `scene: { label, city }` |

引擎（core/flow/map/scenes）**不动**。

## 七、不做的

- 不做真 NLP / 真 LLM 调用（file:// 不可行）
- 不做关卡动态组合（数据池小，成本高收益低）
- 预算/人数不影响实际关卡数据（demo 阶段避免数据膨胀）
- 不改引擎任何逻辑

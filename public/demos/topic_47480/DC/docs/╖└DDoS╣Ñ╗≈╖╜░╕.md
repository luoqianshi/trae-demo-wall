# 秒达定位系统 - 防DDoS攻击方案

## 一、方案概述

### 1.1 背景与目标

秒达定位系统作为国家关键应急基础设施，一旦遭受DDoS攻击导致服务中断，将直接影响紧急呼叫的位置送达，可能造成严重的生命财产损失。因此，必须构建多层次、全方位的DDoS防御体系。

**防御目标：**
- 网络层攻击防护能力：1Tbps+
- 应用层攻击防护能力：100万QPS+
- 正常请求可用性：99.99%
- 攻击检测响应时间：< 10秒
- 误拦截率：< 0.01%

### 1.2 威胁模型

| 攻击类型 | 攻击方式 | 危害等级 | 发生概率 |
|---------|---------|---------|---------|
| SYN Flood | 大量伪造源IP的SYN包 | 🔴 极高 | 高 |
| UDP Flood | 大量UDP数据包 | 🔴 极高 | 中 |
| ICMP Flood | Ping洪水攻击 | 🟠 高 | 中 |
| HTTP Flood | 大量伪造HTTP请求 | 🟠 高 | 高 |
| CC攻击 | 模拟真实用户频繁请求动态页面 | 🟠 高 | 高 |
| DNS放大 | 利用DNS服务器放大攻击流量 | 🟡 中 | 中 |
| NTP放大 | 利用NTP服务器放大攻击流量 | 🟡 中 | 低 |
| 慢速攻击 | Slowloris / Slow POST | 🟡 中 | 低 |
| 应用层DDoS | 针对特定接口的高频请求 | 🟠 高 | 高 |

## 三、核心防护技术

### 3.1 流量清洗（第1-2层）

**技术原理：**
- 流量牵引：将疑似攻击流量引导至清洗中心
- 特征过滤：基于五元组（源IP、目的IP、源端口、目的端口、协议号）过滤
- 速率限制：基于会话的速率限制
- 协议合规：检查数据包是否符合协议规范

**关键指标：**
- 清洗延迟：< 50ms
- 清洗准确率：> 99.9%
- 回切时间：< 30秒

### 3.2 Web应用防火墙（WAF）

**防护规则集：**

| 规则类型 | 规则示例 | 动作 |
|---------|---------|------|
| SQL注入 | `' OR 1=1 --` | 拦截 |
| XSS攻击 | `<script>alert(1)</script>` | 拦截 |
| 命令注入 | `; cat /etc/passwd` | 拦截 |
| 路径遍历 | `../../etc/passwd` | 拦截 |
| 文件上传 | 危险文件类型检测 | 拦截 |
| 恶意爬虫 | 异常User-Agent | 拦截/验证码 |
| 敏感接口 | 非授权访问管理接口 | 拦截 |

### 3.3 限流算法

#### 3.3.1 令牌桶算法（Token Bucket）

```
令牌生成速率：r 个/秒
桶最大容量：b 个令牌

请求到达时：
  - 桶中有令牌 → 消耗1个令牌，放行
  - 桶中无令牌 → 拒绝或排队等待
```

**适用场景：** 接口级QPS限制，允许一定程度的突发流量

#### 3.3.2 漏桶算法（Leaky Bucket）

```
桶容量：b 个请求
出水速率：r 个/秒

请求到达时：
  - 桶未满 → 放入桶中
  - 桶已满 → 拒绝请求
```

**适用场景：** 需要平滑输出流量的场景

#### 3.3.3 滑动窗口计数器

```
时间窗口：T 秒
最大请求数：N

每个请求：
  - 计算当前窗口内请求数
  - 超过N则拒绝
```

**适用场景：** 精确计数的限流场景

### 3.4 IP信誉系统

**黑名单来源：**
1. 实时攻击检测自动封禁
2. 威胁情报订阅（如：微步、AlienVault）
3. 历史攻击数据统计
4. 已知代理/VPN/机房IP段

**分级策略：**
| 风险等级 | 判定条件 | 处置措施 |
|---------|---------|---------|
| 高危 | 明确攻击特征 | 直接封禁 24小时 |
| 中危 | 多次触发限流规则 | 验证码挑战 + 降速 |
| 低危 | 轻微异常行为 | 观察 + 日志记录 |

### 3.5 请求签名验证

**签名生成流程：**
```
1. 拼接字符串：
   stringToSign = HTTPMethod + "\n"
                + Path + "\n"
                + Timestamp + "\n"
                + Nonce + "\n"
                + BodyHash (SHA256)

2. 使用HMAC-SHA256计算签名：
   signature = HMAC-SHA256(secretKey, stringToSign)

3. 放入请求头：
   X-Signature: signature
   X-Timestamp: timestamp
   X-Nonce: nonce
   X-App-Key: appKey
```

**验证逻辑：**
- 时间戳偏差：±300秒
- Nonce去重：5分钟内的随机数不能重复
- 签名匹配：服务端用相同算法计算后比对

## 四、针对紧急定位系统的特殊防护

### 4.1 紧急呼叫接口的特殊性

**核心矛盾：**
- 紧急呼叫不能被误拦截（可能救命）
- 紧急呼叫接口又是攻击的重点目标

**防护原则：**
- 宁漏勿错：宁可放过可疑请求，不能拦截真实紧急呼叫
- 降级有序：攻击发生时按优先级逐步降级，核心功能最后降级
- 多源验证：多维度判断请求合法性，不单一依赖某一特征

### 4.2 分级限流策略

| 接口类型 | 限流粒度 | 默认限流 | 攻击时限流 | 优先级 |
|---------|---------|---------|-----------|-------|
| 紧急位置上报 | 按设备ID | 1次/分钟 | 3次/小时 | P0（最高） |
| 位置查询（接警端） | 按用户ID | 100次/分钟 | 30次/分钟 | P0 |
| 报警状态同步 | 按会话ID | 60次/分钟 | 20次/分钟 | P1 |
| 历史记录查询 | 按用户ID | 20次/分钟 | 5次/分钟 | P2 |
| 统计报表 | 按用户ID | 5次/分钟 | 1次/10分钟 | P3 |
| 管理配置接口 | 按IP白名单 | - | 直接拦截 | P4 |

### 4.3 紧急呼叫白名单机制

**自动白名单：**
- 正在通话中的号码 → 自动加入白名单
- 已确认的有效报警 → 后续请求无限流
- 应急指挥中心IP段 → 永久白名单

**白名单优先级：**
白名单 > 正常限流 > 攻击限流 > 黑名单

### 4.4 服务降级方案

**降级级别定义：**

| 级别 | 触发条件 | 降级措施 |
|------|---------|---------|
| L0（正常） | QPS < 80%阈值 | 全部功能正常 |
| L1（预警） | QPS 80%-100% | 关闭非核心接口的详细日志 |
| L2（轻度） | QPS 100%-150% | 关闭统计报表、历史查询等P3接口 |
| L3（中度） | QPS 150%-300% | 只保留P0、P1核心接口，其余返回降级提示 |
| L4（重度） | QPS > 300% | 只保留紧急位置上报核心功能，其余全挂起 |
| L5（紧急） | 系统即将崩溃 | 只读模式，只接收位置上报，查询排队 |

**降级触发方式：**
- 自动触发：基于实时QPS、CPU、内存、连接数指标
- 手动触发：运维人员确认攻击后手动切换
- 一键降级：SOS按钮，立即进入L4级别

## 五、攻击检测与响应

### 5.1 实时监控指标

**网络层指标：**
- 入站流量速率（bps / pps）
- 新建连接速率
- 并发连接数
- 异常包比例

**应用层指标：**
- 各接口QPS
- 响应时间（P50 / P95 / P99）
- 错误率（4xx / 5xx）
- 限流拒绝数
- 黑名单拦截数

**系统层指标：**
- CPU使用率
- 内存使用率
- 磁盘IO
- 网络IO
- 数据库连接数

### 5.2 异常检测算法

**1. 基线对比法**
```
正常基线 = 历史同时段平均值 × 波动系数

异常判定：
  当前值 > 基线 × 3  →  严重异常
  当前值 > 基线 × 2  →  中度异常
  当前值 > 基线 × 1.5 →  轻度异常
```

**2. 同比环比法**
- 同比：与昨天同一时段对比
- 环比：与上一小时对比
- 偏差超过阈值则告警

**3. 突发检测**
```
滑动窗口：最近5分钟
平均值：avg = sum(window) / len(window)

突发判定：
  当前1分钟值 > avg × 5  →  突发告警
```

## 六、容灾与备份

### 6.1 多活架构

**部署方案：**
- 两地三中心：华北（主）+ 华东（备）+ 华南（灾备）
- 同城双活：同一城市两个机房同时提供服务
- 异地灾备：异地数据实时同步，RTO < 30分钟

**流量调度：**
- DNS智能解析：按地域、运营商、健康状态调度
- GSLB全局负载：实时探测各节点健康，秒级切换
- 单元化架构：每个单元独立完整，故障时只影响部分用户

### 6.2 降级预案清单

| 预案名称 | 触发条件 | 操作步骤 | 预计影响 |
|---------|---------|---------|---------|
| 数据库只读 | 主库CPU > 90% 持续5分钟 | 切换读流量到从库 | 写操作延迟 |
| 缓存扩容 | Redis命中率 < 80% | 临时扩容缓存节点 | 无 |
| 静态资源全缓存 | 出口带宽 > 80% | 提升CDN缓存比例 | 部分内容更新延迟 |
| 非核心接口关闭 | 整体QPS > 200% | 关闭P2/P3接口 | 相关功能不可用 |
| 黑名单全开 | 明确攻击流量 | 批量封禁攻击IP | 可能少量误封 |
| 验证码模式 | 应用层DDoS | 所有非白名单请求需验证码 | 用户体验下降 |

## 七、安全运维规范

### 7.1 应急响应流程

**1. 发现阶段（0-5分钟）**
- 监控系统自动告警
- 值班人员确认告警真实性
- 初步判断攻击类型和规模

**2. 响应阶段（5-30分钟）**
- 启动对应级别应急预案
- 通知相关人员（技术、产品、管理层）
- 同步攻击状态到内部群

**3. 处置阶段（30分钟-2小时）**
- 执行防护策略
- 监控防护效果
- 根据情况调整策略

**4. 恢复阶段（2-24小时）**
- 攻击减弱后逐步恢复服务
- 验证系统完整性
- 保存攻击日志和证据

**5. 复盘阶段（24-72小时）**
- 攻击时间线复盘
- 防护效果评估
- 改进措施制定
- 文档更新

### 7.2 值班制度

- 7×24小时值班制，每班2人
- 5分钟内响应告警
- 重大问题30分钟内升级
- 每日交接班记录

### 7.3 定期演练

- 每月：小规模攻击模拟演练
- 每季度：全链路压测 + DDoS演练
- 每半年：跨机房灾备切换演练
- 每年：真实流量攻防演练（红队蓝队）

## 八、法律与合规

### 8.1 数据留存

- 攻击日志留存：≥ 6个月
- 访问日志留存：≥ 3个月
- 完整数据包捕获（攻击期间）：≥ 30天

### 8.2 证据保全

- 攻击发生时自动开启全流量包捕获
- 关键日志做防篡改处理（区块链/时间戳）
- 必要时申请司法鉴定

### 8.3 追责机制

- 保留追究攻击者法律责任的权利
- 与公安网安部门建立联动机制
- 重大攻击案件及时报案

## 十、方案总结

### 核心设计理念

1. **纵深防御**：多层防护，一层被突破还有下一层
2. **分级处置**：不同攻击级别采用不同策略，避免过度防护
3. **核心优先**：紧急呼叫接口最高优先级，宁可降级也不中断
4. **自动化为主**：90%的攻击由系统自动处理，人工只处理复杂情况
5. **持续进化**：基于攻击数据持续优化规则，越用越聪明

### 关键成功因素

- ✅ 选择有资质的云服务商（具备电信级清洗能力）
- ✅ 提前做好全链路压测，摸清系统极限
- ✅ 制定详细的应急预案并定期演练
- ✅ 建立7×24小时安全运维团队
- ✅ 与公安、运营商建立应急联动机制

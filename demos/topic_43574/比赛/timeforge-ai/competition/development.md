# TimeForge AI - 开发过程记录

## 项目概述

TimeForge AI 是一款面向大学生群体的智能学习任务管理系统，开发周期为 35 天，分为五个阶段逐步推进。以下为各阶段的详细开发记录。

---

## 阶段一：需求分析与技术选型（Day 1-3）

### 主要工作内容

项目启动阶段，团队首先进行了深入的用户调研和竞品分析。通过访谈 20 余名在校大学生，发现普遍存在以下痛点：学习任务繁杂难以统筹管理、DDL 临近才发现任务完不成、缺乏学习规划和动力、学习数据零散无法复盘。基于这些痛点，我们确定了产品的六大核心功能：仪表盘总览、任务管理、AI 任务拆解、DDL 风险预测、学习战力值、数据可视化。

### 关键技术决策

在技术选型上，我们面临"重型框架 vs 轻量方案"的权衡。考虑到比赛场景下评委更关注功能完整性和创新性而非技术复杂度，我们最终选择了 Flask + SQLite + Bootstrap 5 + Chart.js 的轻量技术栈。这一决策的核心考量是：Flask 的极简设计使得 API 开发效率极高，SQLite 零配置的特性省去了数据库部署环节，Bootstrap 5 和 Chart.js 通过 CDN 引入即可获得专业的 UI 和数据可视化效果。

### 遇到的挑战与解决方案

**挑战**：如何平衡功能完整性和开发周期？35 天的时间内需要完成 7 个页面、10+ 个 API 接口和 AI 逻辑引擎的开发。

**解决方案**：采用"核心功能优先、渐进增强"的策略。优先开发仪表盘、任务管理和 AI 拆解三个核心功能，确保 MVP 可用；再逐步追加 DDL 风险预测、战力值、可视化等增强功能。同时，通过预定义模板数据（子任务模板、预估工时映射、AI 建议库）替代复杂的 AI 模型训练，大幅缩短开发时间。

## 阶段二：数据库设计与后端开发（Day 4-10）

### 主要工作内容

这一阶段完成了数据库模型设计、Flask 应用框架搭建和全部 API 接口的开发。数据库设计采用三表结构：users（用户表，存储等级、经验值、连续打卡等数据）、tasks（任务表，存储标题、类型、截止日期、子任务 JSON 等数据）、activities（活动记录表，存储每日专注时长、完成任务数、DDL 风险等数据）。这种设计既满足了功能需求，又保持了结构的简洁性。

### 关键技术决策

**决策一：子任务存储方案**。子任务数据采用 JSON 文本格式存储在 tasks 表的 subtasks 字段中，而非独立的子任务表。这样做的好处是减少了表关联查询的复杂度，因为子任务总是与父任务一起读取和展示。JSON 格式天然支持结构化的子任务数据（标题、完成状态、预估工时），使用 Python 的 json 模块即可轻松序列化和反序列化。

**决策二：AI 引擎设计**。我们没有使用外部 AI API，而是构建了一套基于规则的模拟引擎。核心包括三个部分：SUBTASK_TEMPLATES（子任务模板字典，7 种任务类型各有专属拆解模板）、ESTIMATED_HOURS_MAP（预估工时映射）、ASSISTANT_ADVICE（AI 建议库）。这种设计既保证了 AI 功能的可演示性，又避免了外部 API 调用的不稳定性和成本问题。

### 遇到的挑战与解决方案

**挑战**：DDL 风险预测算法如何设计才能准确反映真实风险？

**解决方案**：经过多轮测试和调优，最终确定了三因子评估模型。以剩余天数、完成率、剩余天数与工时比为三个输入因子，设定了明确的阈值（剩余天数 <= 2 天或完成率 < 30% 触发红色预警）。在后续的种子数据中通过不同截止日期的任务验证了算法的有效性。

## 阶段三：前端界面开发（Day 11-20）

### 主要工作内容

本阶段完成了 7 个页面的前端开发，包括仪表盘首页、任务管理页、AI 任务拆解页、DDL 风险预测页、学习战力值页、AI 课程助手页和数据可视化中心。统一使用 Jinja2 模板继承机制（base.html 作为基础布局），确保页面风格一致、代码复用高效。

### 关键技术决策

**决策一：深色科技风主题**。CSS 变量系统定义了完整的颜色体系（--bg-primary: #0F172A 深蓝黑背景，--color-primary: #6366F1 靛蓝主色），所有组件统一使用这些变量，确保主题一致性。深色主题的另一个优势是降低了长时间使用带来的视觉疲劳感。

**决策二：前端数据获取策略**。每个页面独立调用后端 API 获取数据，使用 Fetch API 异步加载，页面间无数据依赖。这种设计使得每个页面都是独立的功能单元，便于单独调试和维护。JS 文件按页面拆分（dashboard.js 负责仪表盘，charts.js 负责图表），避免代码耦合。

### 遇到的挑战与解决方案

**挑战**：Chart.js 4.x 的配置较为复杂，不同图表类型（热力图、饼图、折线图、雷达图）的配置参数差异大。

**解决方案**：在 charts.js 中封装了统一的图表初始化函数，为每种图表类型预设了默认配置（颜色方案、动画效果、响应式设置），页面只需调用对应函数并传入数据即可生成图表。同时，通过 Chart.js 的全局默认配置统一了字体、颜色和交互行为。

## 阶段四：AI 逻辑与数据可视化（Day 21-28）

### 主要工作内容

这一阶段重点完善了 AI 模拟引擎的逻辑和数据分析可视化功能。为 7 种任务类型分别编写了详细的规划建议（每类 200-300 字），涵盖需求分析、设计、开发、测试、文档等完整流程。数据可视化方面，实现了 30 天热力图、任务完成率饼图、拖延指数仪表盘、专注趋势折线图、DDL 风险变化图、任务类型分布统计和周度学习报告等 7 种可视化形式。

### 关键技术决策

**决策：动态重规划算法的设计**。当用户完成部分子任务后，系统需要根据剩余天数和剩余工时重新生成每日计划。我们设计了一个贪心分配算法：将未完成子任务按天均分，每天分配 ceil(剩余子任务数 / 剩余天数) 个子任务，确保工作量均匀分布。

### 遇到的挑战与解决方案

**挑战**：拖延指数如何量化？

**解决方案**：拖延指数采用"延期任务占比 x 50 + 红色风险任务占比 x 50"的加权计算方式，取值范围 0-100。这个公式兼顾了"已发生的拖延"（延期任务）和"潜在的拖延风险"（红色风险任务），能够较为全面地反映学习者的拖延倾向。通过种子数据中的不同风险等级任务验证了计算结果的合理性。

---

## 阶段五：测试优化与文档（Day 29-35）

### 主要工作内容

最后阶段完成了全面的功能测试、体验优化和文档编写。功能测试覆盖了所有 API 接口的正常响应和异常处理，验证了前端页面的数据加载和交互逻辑。体验优化方面，为所有卡片添加了悬停光影效果，优化了移动端的响应式布局，统一了按钮和表单的交互风格。文档编写包括项目说明（README.md）、部署指南（DEPLOY.md）、作品亮点、开发过程记录、答辩介绍和演示流程。

### 关键技术决策

**决策：演示数据设计**。种子数据的质量直接影响演示效果。我们精心设计了 7 个任务，覆盖了 6 种任务类型和 3 种状态（进行中、已完成、已延期），并设置了不同的截止日期，确保 DDL 风险预测功能能够展示三种风险等级。31 天的活动记录通过随机生成（固定随机种子保证可复现），模拟了真实的学习数据波动。

### 遇到的挑战与解决方案

**挑战**：如何在答辩中有效展示系统功能？

**解决方案**：编写了详细的答辩介绍稿（6 分钟）和演示流程文档（10 步），确保演示过程流畅、重点突出。同时，设计了备用方案（如网络问题时可使用体验版 HTML），降低演示风险。

---

## 总结

TimeForge AI 的开发历时 35 天，总计完成约 1000 行后端代码、1500 行前端代码和 300 行 CSS 样式代码。项目采用"核心功能优先、渐进增强"的开发策略，在有限的时间内实现了完整的学习任务管理闭环。AI 模拟引擎的设计体现了"以规则替代模型"的务实思路，在保证功能可演示性的同时避免了复杂 AI 模型的不确定性。深色科技风的 UI 设计和游戏化战力值系统为产品增添了独特的视觉魅力和用户粘性。

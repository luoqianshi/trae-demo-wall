# 提示词编写技能（Prompt Writing Skill）

**核心定位**：让大模型自己编写结构化提示词，然后**严格按提示词**输出内容。这是把"模型自由发挥"转化为"按指令执行"的关键工具。

## 与 loop engineering 联动（关键）

loop engineering 每轮迭代的 **[2] 写本轮提示词** 步骤必须调用本 skill：

1. 模型先用 prompt-writing skill **设计本轮修复的 5 层结构 prompt**（角色-任务-规则-示例-异常）
2. 然后**严格按 prompt 输出**后续的修复代码和说明
3. 把"凭直觉改代码"升级为"按预设指令执行"，避免发散

**反例**（不调用本 skill 的后果）：
- 6 轮迭代只跑 1 轮，每轮聚焦什么完全凭感觉
- 修复方向不统一，有时重构有时只改一行
- 输出格式混乱，团队无法复用

## 一、5 层结构模板

| 层级 | 名称 | 作用 | 示例 |
|------|------|------|------|
| L1 | 角色定义 | 明确 AI 身份与能力边界 | "你是一位资深跑步训练教练" |
| L2 | 任务描述 | 说清楚要做什么 | "本轮修复 VDOT 配速不现实问题" |
| L3 | 规则约束 | 限定输出格式、范围、禁区 | "必须用 Daniels 官方表插值，不编造数据" |
| L4 | 示例引导 | 给出期望输出范例 | "VDOT=50 → E:4:58, M:4:33, T:4:00" |
| L5 | 异常处理 | 明确边界场景行为 | "VDOT<30 走表下限，>80 走表上限" |

### 完整模板

```
[角色 L1] 你是一个____，擅长____。
[任务 L2] 请____，目标是____，本轮聚焦____维度。
[规则 L3]
  1. ____
  2. ____
  3. 禁止：____
[示例 L4] 输入____ → 输出____
[异常 L5] 如果____，则____
```

### Loop Engineering 中的具体应用

每轮迭代开始时，模型应先输出这样的 prompt：

```markdown
## 本轮提示词

**角色 L1**：你是一个____工程师，正在 CoachEngine 项目做 loop engineering 第____轮
**任务 L2**：修复____问题
**规则 L3**：
- 必须用____公式（来源：____）
- 不得修改____（SSOT 强制）
- 函数长度 ≤ 20 行
- 中文注释解释"为什么"而非"做什么"
**示例 L4**：
- 输入 VDOT=50 → 输出 E 配速 4:58/km
- 输入 startDate=周三 → 输出 residualDays=2
**异常 L5**：
- VDOT < 30 → 走表下限
- 边界值异常 → 提前 return
```

然后**严格按上述 prompt 执行**后续的 Read/Edit/验证。

## 二、场景适配方法

### 场景 A：算法实现类
**重点**：公式来源、参数单位、边界值、验证用例

```
[规则]
- 公式来源：Jack Daniels《Daniels' Running Formula》
- 参数单位：速度 v 为 m/min，时间 t 为 min
- 边界值：VDOT < 30 取表下限，VDOT > 80 取表上限
- 验证：VDOT=50 时 E 配速应为 4:58/km（±1s）
```

### 场景 B：UI 渲染类
**重点**：状态分支、空状态、交互事件绑定

```
[规则]
- 状态分支：PAST / REST / 训练日 / 比赛日 分别渲染
- 空状态：无计划时显示"请先到生成页创建计划"
- 事件：按钮必须 event.stopPropagation() 防冒泡
```

### 场景 C：迭代调试类
**重点**：聚焦维度、修复前先读、验证方式

```
[规则]
- 本轮聚焦：状态管理与持久化
- 修复前必须先 Read 文件，不凭记忆改
- 修复后必须浏览器实测验证
```

## 三、优化策略

### 策略 1：分步拆解
复杂任务拆成子任务，每个子任务单独提示：
```
❌ "生成完整训练计划并实现 UI 和交互"
✅ Step1: "实现 VDOT 计算函数" → 验证 →
   Step2: "实现 12 周课表生成" → 验证 →
   Step3: "实现 UI 渲染"
```

### 策略 2：负面约束
明确"不要做什么"比"要做什么"更有效：
```
[禁止]
- 不要在生成计划时预填 completed/skipped 状态
- 不要在 UI 渲染中使用未定义的 CSS 变量
- 不要用 oninput 触发整页 reRender（IME 闪烁问题）
```

### 策略 3：验证锚点
在提示词中嵌入可验证的锚点：
```
[验证锚点]
- 残周模式 startDate=2026-06-10（周三）→ residualDays=2
- VDOT=50 → E 配速 300 秒/km = 5:00/km
- 完成硬课后如果明天也是硬课 → 必须下推
```

### 策略 4：上下文传递
跨轮次迭代时，传递关键上下文：
```
[历史修复记录]
- R1: VDOT 配速不现实 → 改用 Daniels 表插值
- R2: 残周索引错位 → 引入 absoluteDay 算法
[本轮聚焦] R3: 状态管理与持久化
```

## 四、任务类型模板库

### 模板 1：新功能开发
```
[角色] 你是一个____工程师
[任务] 实现____功能
[输入] ____
[输出] ____
[规则] 遵循项目现有架构，不引入新依赖
[验证] ____
```

### 模板 2：Bug 修复
```
[现象] ____（具体表现）
[复现] ____
[根因分析] 请先分析可能原因，不要直接改
[修复约束] 只改必要部分，不重构周边代码
[验证] 修复后验证____
```

### 模板 3：代码审查
```
[审查范围] ____
[审查维度] 逻辑正确性 / 状态管理 / 性能 / 安全
[输出格式] 问题清单 + 严重度 + 修复建议
[禁止] 不直接改代码，只提建议
```

### 模板 4：迭代自审
```
[本轮维度] ____
[扫描范围] ____
[检查清单] ____
[修复原则] 小步迭代，改完即验证
[终止条件] 连续一轮无新问题
```

## 五、质量检查清单

提示词编写完成后，逐项检查：

- [ ] 角色定义是否清晰（AI 知道自己是谁）
- [ ] 任务描述是否具体（AI 知道要做什么）
- [ ] 规则约束是否完整（AI 知道边界在哪）
- [ ] 是否有验证锚点（输出可验证）
- [ ] 是否有异常处理（边界场景覆盖）
- [ ] 是否有负面约束（禁止项明确）
- [ ] 复杂任务是否已拆解（分步执行）

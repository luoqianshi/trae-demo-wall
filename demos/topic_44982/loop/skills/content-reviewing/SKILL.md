# 内容审查技能（Content Reviewing Skill）

建立全面的内容质量评估体系，对 AI 输出和代码内容进行多维度审查。

## 与 loop engineering 联动

loop engineering 每轮迭代的 [I] 报告质量审查 步骤必须调用本 skill，重点审查：
- **准确性**：修复报告中的 Bug 描述、修复方案、验证结果是否与代码实际状态一致
- **逻辑性**：报告中的因果链是否清晰（症状→根因→修复→预防）
- **完整性**：是否漏掉副作用评估、回归测试、相关文件清单
- **可读性**：是否对所有修复项分轮次、分类别组织，是否便于团队 review

**反例**：修复报告只列"修复了 X"，没说 X 是什么、为什么坏、怎么修，缺根因。

## 一、审查维度矩阵

### 维度 1：准确性（Accuracy）
**定义**：内容是否与客观事实、权威标准一致

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| 公式正确性 | 对照原始论文/官方文档 | 参数单位、系数完全一致 |
| 数据真实性 | 查验数据来源 | 不编造不存在的数据 |
| 常识判断 | 用领域常识验证 | 结果在合理范围内 |
| 引用准确性 | 核实引用来源 | 引用内容与原文一致 |

**准确性审查清单**：
```
□ 公式来源是否标注（如 Daniels《Running Formula》）
□ 参数单位是否一致（m/min vs km/h vs sec/km）
□ 边界值是否合理（VDOT < 30? 心率 < 0?）
□ 验证锚点是否通过（VDOT=50 → E:4:58/km）
```

### 维度 2：逻辑性（Logicality）
**定义**：推理链条是否完整，因果关系是否成立

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| 条件分支完整性 | 枚举所有输入情况 | 每个分支都有处理 |
| 因果链正确性 | 追踪数据流 | 无循环依赖、无死路径 |
| 状态一致性 | 检查状态转换 | 无非法状态跳转 |
| 边界处理 | 极值测试 | 0/负数/空值/超大值不崩溃 |

**逻辑性审查清单**：
```
□ if/else 是否覆盖所有情况（含 else if 链末端）
□ 循环是否有终止条件（防死循环）
□ 状态转换是否合法（PAST→不能直接complete）
□ 边界值：dayIndex=0, dayIndex=6, weekIndex=0, weekIndex=11
```

### 维度 3：专业性（Professionalism）
**定义**：是否符合领域最佳实践和行业规范

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| 术语准确性 | 领域专家审查 | 使用正确专业术语 |
| 方法论合理性 | 对照行业标杆 | 采用公认方法论 |
| 参数合理性 | 文献佐证 | 参数在文献推荐范围内 |
| 可解释性 | 非专家能否理解 | 关键决策有注释说明 |

**专业性审查清单**：
```
□ 心率区间是否符合运动生理学（Karvonen 公式 vs 随意偏移）
□ 训练分区是否符合 ACSM 指南
□ 配速表是否与 Daniels 官方一致
□ 伤病风险阈值是否有文献支撑
```

### 维度 4：安全性（Safety）
**定义**：是否存在可能导致数据丢失、安全漏洞或人身伤害的风险

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| 输入校验 | 注入测试 | 恶意输入不执行 |
| 数据保护 | 权限审查 | 敏感数据不泄露 |
| 物理安全 | 训练建议审查 | 不给出可能导致受伤的建议 |
| 错误处理 | 异常路径测试 | 错误不导致崩溃 |

**安全性审查清单**：
```
□ 用户输入是否经过校验（parseTimeToSec 拒绝非数字）
□ 是否有 XSS 风险（innerHTML 拼接用户输入）
□ 训练建议是否安全（不推荐超出能力范围的训练）
□ 错误状态是否优雅降级（不白屏不卡死）
```

## 二、审查流程

### 标准审查流程（6 步法）

```
Step 1: 确定审查范围
  → 明确审查哪个文件/模块/功能

Step 2: 选择审查维度
  → 根据内容类型选择重点维度
  → 算法代码：准确性 + 逻辑性
  → UI 代码：逻辑性 + 安全性
  → 文档内容：准确性 + 专业性

Step 3: 静态审查
  → Read 精读关键代码
  → Grep 搜索已知坏味道模式
  → 列出疑似问题清单

Step 4: 动态验证
  → 浏览器实测核心流程
  → 控制台检查 JS 错误
  → 边界值输入测试

Step 5: 问题分级
  → P0 致命：崩溃/数据丢失/安全漏洞
  → P1 严重：功能错误/逻辑缺陷
  → P2 一般：体验问题/代码不规范
  → P3 建议：优化建议/风格改进

Step 6: 输出审查报告
  → 问题清单 + 分级 + 修复建议
```

### 审查报告模板

```markdown
## 内容审查报告

**审查范围**：____
**审查日期**：____
**审查维度**：____

### 问题清单

| # | 维度 | 严重度 | 问题描述 | 位置 | 修复建议 |
|---|------|--------|----------|------|----------|
| 1 | 准确性 | P0 | VDOT 计算结果偏离 | L350 | 公式参数错误 |
| 2 | 逻辑性 | P1 | 残周索引边界未处理 | L520 | 添加 absoluteDay |
| 3 | 安全性 | P2 | innerHTML 拼接用户输入 | L840 | 改用 textContent |

### 审查结论
- 通过 / 需修复后通过 / 需返工
- 核心风险：____
- 改进建议：____
```

## 三、坏味道模式库

### 逻辑类坏味道

| 模式 | 症状 | 检查方法 |
|------|------|----------|
| 悬空 else | if 链没有 else 兜底 | 搜索 `if(` 检查是否有 else |
| 隐式全局变量 | 未声明直接赋值 | 严格模式检查 |
| 状态泄漏 | 编辑态污染显示态 | 检查 editing 标志 |
| 幂等性缺失 | 重复调用结果不同 | 模拟连续点击两次 |

### 安全类坏味道

| 模式 | 症状 | 检查方法 |
|------|------|----------|
| innerHTML 注入 | 用户输入直接拼入 HTML | Grep `innerHTML.*input` |
| eval 陷阱 | 动态执行字符串 | Grep `eval(` |
| 原型污染 | 对象属性被篡改 | 检查 `__proto__` 访问 |
| 不安全的 JSON | 未校验的 JSON.parse | 检查 try-catch 包裹 |

### 性能类坏味道

| 模式 | 症状 | 检查方法 |
|------|------|----------|
| 整页 reRender | 点击触发改重建 DOM | 检查是否可用局部更新 |
| 循环内重复计算 | 每次迭代重算相同值 | 提取到循环外 |
| 不必要的对象创建 | 循环内 new 对象 | 复用对象 |
| 事件监听泄漏 | 未清理的事件绑定 | 检查 removeEventListener |

## 四、审查触发条件

- 每轮迭代完成后：必须执行审查
- 新功能开发完成：必须执行审查
- 用户明确要求审查时：立即执行
- 修复 Bug 后：对修复点及关联模块执行审查
- 发布前：全维度审查

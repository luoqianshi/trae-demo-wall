# 双账户转账

## 背景

`Bank` 是一个并发银行转账系统，构造时创建 `n` 个账户，每个账户初始余额为 `initial`，
因此系统总额恒为 `n * initial`。其公开 API：

- `Bank(int n, long initial)`：创建 n 个账户，每个初始余额 initial。
- `void transfer(int from, int to, long amt)`：从 `from` 账户扣 `amt`、给 `to` 账户加 `amt`；
  `from == to` 时为空操作。该接口允许被多个线程并发调用。
- `long balance(int i) const`：返回第 i 个账户的余额。
- `long total() const`：返回所有账户余额之和。
- `void run_transfers(int n_threads, int per_thread, int n_accounts, long amt)`：
  辅助方法，启动 `n_threads` 个线程，每个线程在 `n_accounts` 个账户间做 `per_thread` 次转账（金额 amt）。

## 契约

- **资金守恒**：转账只是把钱从一个账户搬到另一个账户，系统总额永不改变。
  任意时刻——包括在其它线程的 `transfer` 正在进行中——调用 `total()`，
  结果都必须严格等于初始总额 `n * initial`。
- **并发安全**：在任意并发转账组合下都不得丢失更新，也不得凭空生钱或丢钱。

## 现象

系统在单线程、低负载下运行完全正常：逐笔转账后对账都精确平衡。

但在高并发转账压力下，监控线程偶尔会发现账目对不上——汇总出来的总额一会儿偏多、
一会儿偏少，仿佛钱在转账途中"凭空消失又出现"。诡异的是，等所有转账线程都结束、
系统静止后再去汇总，总额又是精确正确的；单线程顺序跑也从不出问题。
并发度越高、互相对转越密集，越容易观察到这种瞬时的账目不平衡。

## 任务

定位并修复实现，使其在**任意并发转账组合**下资金都严格守恒：
即便在转账进行中调用 `total()`，也必须始终等于初始总额。请勿改动公开 API。

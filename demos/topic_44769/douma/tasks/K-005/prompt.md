# 批量部件注册表

## 背景

`Registry` 是一个部件（`Widget`）注册表：业务以**批次**为单位提交一组部件 id，注册表为每个 id
在堆上创建一个 `Widget` 并纳入管理。每个 `Widget` 都是受管理的资源——它在构造时占用资源、
析构时释放，全局存活实例数由 `Widget::alive` 精确反映（用于泄漏检测）。

批次中可能混入**非法 id**（本题约定 `id < 0` 为非法）。一旦发现非法 id，整批必须被拒绝：
抛出异常，并保证「**要么整批成功纳入，要么注册表完全不变**」（强异常安全），
且**绝不能泄漏**任何在处理这一批过程中已经创建出来的 `Widget`。

## 对外接口

- `Registry()` / `~Registry()`：创建 / 销毁注册表（销毁时释放其管理的全部 `Widget`）。
- `void add_batch(const std::vector<int>& ids)`：把一批 id 对应的部件纳入注册表。
  - 全部 id 合法 → 全部纳入，`size()` 增加 `ids.size()`。
  - 含非法 id（`id < 0`）→ 抛出 `std::exception` 派生异常，注册表**保持调用前的状态不变**。
- `std::size_t size() const`：当前管理的部件数量。
- `struct Widget`：含静态成员 `static long alive`（当前存活的 Widget 实例数）。

## 对外保证（不变量）

1. **强异常安全**：`add_batch` 若抛出异常，注册表状态与调用前**完全一致**（`size()` 不变），
   不得出现「半提交」——部分纳入、部分丢弃。
2. **无泄漏**：任何一次 `add_batch`（无论成功还是抛异常）结束后，
   `Widget::alive` 必须恰好等于「当前注册表中管理的 Widget 数」——
   异常路径上已临时创建的 Widget **必须全部被释放**，`alive` 不得只增不减。
3. **析构归零**：`Registry` 销毁后，它曾管理的 Widget 全部释放。

## 错误现象

- 提交的批次全部合法时，注册与计数完全正常，`alive` 与 `size()` 始终吻合。
- 但只要某个批次中含有非法 id 触发异常，虽然异常被正确抛出、注册表大小也没变，
  `Widget::alive` 却**比实际管理的数量多**——而且每经历一次失败批次就多一些，
  反复失败后存活计数持续攀升，说明异常路径上创建的部件没有被释放。

定位并修复缺陷，使上述三条不变量在任意批次（含非法 id、反复失败）下都严格成立。

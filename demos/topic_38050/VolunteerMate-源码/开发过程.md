# 📘 开发过程与踩坑记录

> 本文档详细记录了 VolunteerMate 从 0 到 1 的开发过程，包含使用的 TRAE 能力、关键提示词和遇到的坑。

## 🎯 项目启动与架构决策

### 1. 选题过程

> **使用能力**：TRAE 的代码理解 + 文档生成

**最初的构思**：想做一个"AI 工具类"产品，但发现竞品太多。
**探索过程**：
- 尝试过 AI 学习助手 —— 功能太泛，缺少差异化
- 尝试过 AI 生活助手 —— 场景不够具体
- 尝试过公益打卡 —— 简单但完整，适合做 demo

**最终选定理由**：
1. 公益主题正面，有社会价值，容易引起共鸣
2. "打卡"这个模式天然适合做前端应用，功能边界清晰
3. 可以自然地融入多个 AI 能力（推荐、问答、画像分析、生成）

### 2. 技术选型

> **使用能力**：TRAE 的项目脚手架 + 依赖管理

**决策过程**：
- ✅ **React 18 + TypeScript** —— 组件化开发，类型安全，适合团队项目
- ✅ **Tailwind CSS** —— 快速搭建 UI，易于控制视觉一致性，减少 CSS 文件数量
- ✅ **Zustand** —— 比 Redux 轻量 10 倍，API 简洁，内置 devtools
- ✅ **Recharts** —— 成熟稳定的图表库，组件化 API，支持饼图/雷达图/折线图
- ✅ **Vite** —— 开发体验极致，HMR 秒开
- ✅ **单文件输出** —— 便于提交到比赛（只需上传一个 HTML）
- ❌ **未使用 UI 框架**（如 Ant Design / shadcn）—— 考虑到比赛作品的原创性展示，选择手搓组件

### 3. 项目结构设计

```
src/
├── ai/              ← AI 业务逻辑（与 UI 完全解耦）
├── components/      ← 可复用组件（每个组件单一职责）
├── data/            ← 数据模型 + Mock 数据
├── store/           ← 全局状态（Zustand）
├── pages/           ← 页面路由
├── hooks/           ← 自定义 hooks
└── lib/             ← 工具函数
```

**设计原则**：数据驱动 UI，逻辑层与视图层完全分离，便于后续升级替换。

## 🤖 AI 功能设计过程

### 1. AI 助手模块

> **使用能力**：TRAE 的代码生成 + API 调用辅助

**第一轮**：只做了规则问答 —— 体验差，像搜索引擎
```typescript
// 早期版本：关键词匹配
const answerPool = {
  "环保": "环保可以从垃圾分类开始...",
  "捐赠": "捐赠前请先整理物品..."
}
```
问题：回答生硬，无法处理复杂问题，很容易暴露"不是真 AI"

**第二轮**：接入真实 API —— 支持自由对话
```typescript
// 采用 OpenAI 兼容接口
fetch("https://api.deepseek.com/chat/completions", {
  headers: { Authorization: `Bearer ${key}` },
  body: JSON.stringify({
    model: "deepseek-chat",
    messages: [...],
  })
})
```
改进：回答自然，能处理任意问题，但有两个新坑
- 用户需要配置 API Key —— 增加了使用门槛
- Key 直接写在前端可能会泄露 —— 改为让用户自行配置

**第三轮**（最终方案）：双轨架构
- 有 API Key → 真实对话
- 无 API Key → 规则式回答 + 明确提示"可配置 AI"

### 2. AI 推荐引擎

> **使用能力**：TRAE 的逻辑分析 + 算法辅助设计

**设计思路**：做一个"轻量推荐系统"
```
输入: (用户画像, 当日主题, 时间段, 历史打卡记录)
↓
打分: 每个任务根据匹配因素计算得分
↓
输出: Top 3 推荐任务 + 个性化推荐理由
```

**匹配因子**：
1. 类型偏好评级（用户做过的任务类型）
2. 新任务探索奖励（从未做过的类型 +5 分）
3. 连续激励（连续 3 天同类 +8 分）
4. 难度适配（用户等级高，推荐更难任务）
5. 今日主题加成（+15 分）

### 3. 公益画像雷达

> **使用能力**：TRAE 的数据可视化方案设计

**六维评估指标**：环保 / 捐赠 / 帮扶 / 传播 / 坚持 / 时长

### 🔴 坑 1：Vite 构建时 TS 类型报错 — localStorage 读取的 string 无法赋值给联合类型

**问题**：`src/ai/apiService.ts` 中 `localStorage.getItem('ai_provider')` 返回 `string | null`，但 `APIProvider` 是 `'deepseek' | 'siliconflow' | 'dashscope' | 'zhipu'` 联合类型，TS 严格模式下报错 `Type 'string' is not assignable to type 'APIProvider'`。

```typescript
// ❌ 报错
const providerId = localStorage.getItem('ai_provider') || 'siliconflow';

// ✅ 修正：用 as 强制断言 + fallback
const providerId = (localStorage.getItem('ai_provider') as APIProvider) || 'siliconflow';
```

**💡 解决**：所有从 localStorage 读取的枚举值，都用 `as 类型断言` + `|| 默认值` 双层保护。

**🎯 启示**：类型问题不要用 `any` 绕过，越绕越乱。花 5 分钟把类型写好，后续开发事半功倍。

### 🔴 坑 2：Recharts 图表在移动端的尺寸自适应问题

**问题**：雷达图/饼图在 375px iPhone SE 上完全空白，或者溢出容器。在 PC 上正常，但移动端就崩。

**排查**：
1. 加红边框检查父容器尺寸 —— 容器正常，但图表 SVG 的 viewBox 算错
2. 发现 Recharts 默认用 px 固定宽高，不响应 `width: 100%`
3. 外层容器 `overflow: hidden` 会裁切掉坐标轴标签

**💡 解决**：
```tsx
<div className="w-full" style={{ height: 240 }}>
  <ResponsiveContainer width="100%" height="100%">
    <RadarChart data={...} outerRadius="70%">
      ...
    </RadarChart>
  </ResponsiveContainer>
</div>
```
关键：使用 `ResponsiveContainer` 包裹，外层给固定高度，内部自适应宽度；`outerRadius` 用百分比而非 px。

**🎯 启示**：图表库的"响应式"通常不是开箱即用的，需要用容器约束。不要假设库帮你处理了所有 edge case。

### 🔴 坑 3：localStorage 在打包后数据格式的兼容问题

**问题**：开发模式（`npm run dev`）下打卡记录正常，但 build 后用 `file://` 打开，刷新页面后数据"丢了一半"。

**排查发现**：
- Zustand persist middleware 会把 state 序列化为 JSON 存入 localStorage
- 首次启动时 `localStorage.getItem('volunteer-mate-state')` 为 `null`，JSON.parse(null) 也是 `null` 不是报错
- 但部分字段（如 `Date` 类型）JSON 序列化后变成 string，反序列化时没有恢复成 Date，导致时间计算出错

**💡 解决**：
```typescript
// 方案 A：在 store 初始化时，对 null 值做保护
const initialStats: UserStats = {
  totalMinutes: 0,
  totalCheckIns: 0,
  currentStreak: 0,
  // ... 每个字段都显式给默认值，绝不让它 undefined
};

// 方案 B：对时间戳字段，在读取时手动重新解析
const checkIns = rawCheckIns.map((c) => ({
  ...c,
  timestamp: typeof c.timestamp === 'number' ? c.timestamp : new Date(c.timestamp).getTime(),
}));
```

**🎯 启示**：`file://` 协议下 localStorage 的可用额度、行为和 `http://` 不完全一致。**一定要在真实部署环境下测试**，不能只测 dev server。

### 🔴 坑 4：Tailwind 动态 class 名（如 `bg-${color}-500`）在 build 后丢失

**问题**：开发模式颜色正常，但 build 后动态设置的背景色全白了。

**根本原因**：Tailwind 是按需编译的（JIT 模式），它只编译源码中**完整出现**的 class 名。`bg-${color}-500` 这种模板字符串，Tailwind 的静态扫描器识别不出来，不会把对应的 CSS 规则打入产物。

```tsx
// ❌ build 后 bg-xxx-500 的样式根本没生成
<div className={`bg-${color}-500`}>

// ✅ 用 CSS 变量或完整 class 映射
const colorMap: Record<string, string> = {
  '环保': 'bg-emerald-500',
  '捐赠': 'bg-pink-500',
  '帮扶': 'bg-purple-500',
  '传播': 'bg-blue-500',
};
<div className={colorMap[category]}>
```

**💡 解决**：所有动态 class，要么用映射表（完整写出每个 class 字符串），要么在 `tailwind.config.js` 的 `safelist` 里手动声明。

**🎯 启示**：Tailwind 的"魔法"是有前提的——静态可扫描。任何需要运行时拼接的 class，都会在 build 时被砍掉。

### 🔴 坑 5：移动端触摸反馈与按钮的防抖问题

**问题**：在手机上快速连点"打卡"按钮，会触发多次 addCheckIn，导致同一条任务被重复记录，成就检测也被触发多次。

**排查**：
- 触摸事件 `onTouchStart` + `onClick` 在移动端可能同时触发
- 庆祝动画弹窗有 300ms 延迟，期间按钮仍可点击

**💡 解决**：
```tsx
const [isCheckingIn, setIsCheckingIn] = useState(false);

const handleCheckIn = useCallback(() => {
  if (isCheckingIn) return;           // 防抖第一层
  setIsCheckingIn(true);
  // ... 执行打卡逻辑
  setTimeout(() => setIsCheckingIn(false), 800);  // 800ms 后解锁
}, [isCheckingIn]);

<button disabled={isCheckingIn} onClick={handleCheckIn}>
  {isCheckingIn ? '已打卡 ✓' : '立即打卡'}
</button>
```

**🎯 启示**：移动端的点击延迟和误触是真实存在的问题。所有"触发数据变更"的按钮，都需要**防抖 + disabled 状态**双重保护。

### 🔴 坑 6：CORS 与 API 请求在 file:// 协议下的限制

**问题**：build 后的 index.html，双击打开用 `file://` 协议访问，AI 对话全部失败。控制台报错 `Failed to fetch` / `URL scheme must be "http" or "https"`。

**根本原因**：浏览器对 `file://` 协议的安全策略非常严格——不允许发起跨源（甚至同文件内的）fetch 请求。虽然我们调用的是 `https://api.siliconflow.cn` 的 HTTPS 接口，但在 `file://` 页面上发起请求就会被拦截。

**💡 解决**：
1. 在 vite.config.ts 中设置 `base: './'`，确保产物可以用相对路径加载
2. 接入 `vite-plugin-singlefile`，把 JS/CSS 全部内联到一个 HTML
3. **关键**：在 AI 助手页添加检测逻辑，若检测到 `file://` 协议，提示用户"用本地 HTTP 服务器打开"或切换到"规则问答模式"
4. 硅基流动等现代 API 通常返回 `Access-Control-Allow-Origin: *`，在 http/https 页面下可以正常请求

**🎯 启示**：比赛评审很可能直接双击 HTML 文件打开。**file:// 协议的兼容性必须单独测试**，不要假设"浏览器能打开就能跑"。

### 🔴 坑 7：Zustand store 在组件间初始化时序问题

**问题**：个人中心页面的雷达图第一次渲染时数据为 0，需要切到其他 tab 再切回来才显示正确数据。

**排查**：
- HomePage 组件在打卡时调用 `useStore.getState().addCheckIn(task)`
- ProfilePage 组件用 `useStore((s) => s.stats)` 订阅
- 但 ProfilePage 在 HomePage 之前挂载（因为是单页应用，所有页面都已初始化）
- 当首页更新 store 后，ProfilePage 的 useEffect 依赖了旧值

**💡 解决**：
```tsx
// ❌ 依赖旧值，不触发 re-render
const stats = useStore((s) => s.stats);
useEffect(() => { renderChart(stats); }, []);  // 空依赖

// ✅ 直接订阅 selector，依赖项里加上 stats
const stats = useStore((s) => s.stats);
useEffect(() => { renderChart(stats); }, [stats]);

// 更好的方案：用 shallow 比较
const stats = useStore((s) => s.stats, shallow);
```

**🎯 启示**：全局状态管理 + 多页面共享时，**每个组件都要显式声明它依赖的 state 片段**，不要假设"mount 时已经有最新值"。Zustand 的订阅机制是严格的——你不说你依赖谁，它就不通知你。

### 🔴 坑 8：超长文本在卡片内溢出的处理

**问题**：任务描述和 AI 推荐理由是用户自定义长度的文案，在某些屏幕宽度下，文字从卡片右侧溢出，破坏了圆角卡片的视觉效果。

**尝试 1**：`overflow: hidden` —— 文字被裁断，体验差
**尝试 2**：`word-break: break-all` —— 中文断词位置奇怪
**尝试 3**：`text-overflow: ellipsis` + 固定行数

**💡 最终方案**（Tailwind + CSS）：
```tsx
<div className="text-sm text-gray-600 leading-relaxed overflow-hidden"
     style={{
       display: '-webkit-box',
       WebkitLineClamp: 3,
       WebkitBoxOrient: 'vertical',
     }}>
  {description}
</div>
```
配合外层 `min-h-0`（Flex 容器内的关键属性），解决了卡片被内容撑破的问题。

**🎯 启示**：中文文本的溢出处理比英文复杂，因为没有空格作为天然断词点。`-webkit-line-clamp` 虽然是 webkit 前缀，但在现代浏览器（包括移动端）兼容性很好，是处理多行省略的最佳方案。

### 🔴 坑 9：AI 响应的流式/非流式切换设计

**问题**：最初想用 SSE（Server-Sent Events）流式输出，让 AI 打字效果更真实。但发现：
1. 不同 API 提供商的流式实现不一致（OpenAI 用 `data: [DONE]`，某些提供商用不同的结束标记）
2. `file://` 协议下 fetch 的 stream reader 行为不可靠
3. 比赛作品的 AI 功能应该是"能跑就行"，不是"做一个聊天平台"

**💡 解决**：**统一用非流式模式**，前端模拟打字效果：
```typescript
const response = await fetch(provider.url, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', 'Authorization': `Bearer ${apiKey}` },
  body: JSON.stringify({
    model: provider.defaultModel,
    messages: [...],
    stream: false,        // ← 非流式，一次性拿到完整回答
    max_tokens: 800,
    temperature: 0.7,
  }),
});

// 前端模拟打字：每隔 30ms 追加一个字，制造"AI 正在思考"的感觉
const fullText = data.choices[0].message.content;
for (let i = 0; i < fullText.length; i++) {
  await new Promise((r) => setTimeout(r, 30));
  setDisplayText(fullText.slice(0, i + 1));
}
```

**🎯 启示**：工程决策要权衡"真实效果"和"实现成本"。对比赛作品，**非流式 + 前端模拟**是性价比最高的方案——看起来像流式，实现成本只有 1/10。

### 🔴 坑 10：多个动画效果的性能优化（粒子效果 + 渐变色）

**问题**：打卡庆祝动画用了 60 个粒子 + `requestAnimationFrame` + 渐变色背景，在低端 Android 手机上明显卡顿，FPS 掉到 30 以下。

**排查**：
- 粒子每个都用独立的 div + transform，每帧重排 60 个 DOM
- 渐变色背景用 `linear-gradient` + `animation`，CSS 动画每帧触发 paint
- 叠加 `backdrop-filter: blur()` 毛玻璃效果，浏览器需要对每个像素做高斯模糊

**💡 解决**：
```tsx
// 优化 1：粒子数量根据屏幕宽度动态调整（手机端 30 个，平板 60 个）
const particleCount = window.innerWidth < 480 ? 30 : 60;

// 优化 2：用 transform: translate3d() 开启硬件加速
// translate3d 触发 GPU 合成层，避免每帧重排
style.transform = `translate3d(${x}px, ${y}px, 0)`;

// 优化 3：毛玻璃效果只在"性能好"的设备上启用
// 用 prefers-reduced-motion 检测用户偏好
const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
if (prefersReducedMotion) {
  particleCount = 0;  // 尊重"减少动态效果"的设置
}

// 优化 4：动画结束后清空粒子容器的 DOM，释放内存
useEffect(() => {
  return () => { /* 卸载时清理 */ };
}, []);
```

**🎯 启示**：动画效果的"视觉冲击力"和"性能"是线性关系。**在低配设备上砍掉一半粒子数量，比优化代码更有效**。永远给用户（和浏览器）一个"降级体验"的选项。

### 🔴 坑 11：单文件打包后图片 base64 大小限制

**问题**：`vite-plugin-singlefile` 默认把小于 4KB 的资源内联为 base64，大于 4KB 的仍然作为外部文件。导致 build 后产物里出现额外的 `.png` 文件，不能真正"单文件部署"。

**排查**：
- 检查 `dist/index.html` —— 发现还有 `<img src="assets/logo-xxx.png">`
- vite-plugin-singlefile 的 `inlineFileSizeLimit` 默认值是 4096 bytes
- 我们的 emoji 图标 + 渐变背景没有图片，但 lucide-react 某些图标打包后超过了这个阈值

**💡 解决**：
```typescript
// vite.config.ts
export default defineConfig({
  base: './',
  build: {
    target: 'esnext',
    sourcemap: false,
    assetsInlineLimit: 100 * 1024,   // ← 100KB 以内都内联
  },
  plugins: [
    react(),
    viteSingleFile({
      removeViteModuleLoader: true,
    }),
  ],
});
```
配合 `lucide-react` 的按需导入（`import { Home } from 'lucide-react'` 而不是整包 import），确保图标是 SVG 不是 bitmap。

**🎯 启示**：单文件打包不是"装上插件就完事"。`vite-plugin-singlefile` 依赖于 Vite 本身的 `assetsInlineLimit`，两个配置要一起调。**build 后一定要用文件管理器验证——真的只有一个 HTML 文件吗？**

### 🔴 坑 12：深浅色配色下的可读性问题

**问题**：在某些低亮度手机屏幕（或开启"降低白点值"的 iPhone）上，浅灰色文字 + 渐变色背景的对比度不够，文字看不清。

**排查**：
- 用 Chrome DevTools 的 Lighthouse 检查 Accessibility —— 发现 3 处文字对比度低于 WCAG AA（4.5:1）
- `text-gray-500` 在渐变色背景上有时对比度只有 2.5:1
- 毛玻璃效果 `backdrop-blur` 让背景文字和前景文字对比度进一步下降

**💡 解决**：
```tsx
// 1. 正文文字至少用 text-gray-700（#374151），不用 text-gray-500
// 2. 渐变色卡片上的文字，统一用白字 + text-shadow 增强
<span className="text-white font-medium" style={{ textShadow: '0 1px 2px rgba(0,0,0,0.2)' }}>

// 3. 毛玻璃效果的卡片，背景加半透明不低于 0.8 的 alpha
<div className="backdrop-blur-md bg-white/85">  // ← bg-white/85 比 bg-white/60 可读性好

// 4. 图表用深色主题配色（#0f172a, #0ea5e9, #8b5cf6），确保在白底上可见
<Cell fill="#0f172a" />  // slate-900，不是 slate-500
```

**🎯 启示**：**不要只在你的 4K 高分屏上看效果**。用 DevTools 的 Lighthouse / Accessibility Inspector 跑一遍，或者在真实的低亮度手机上看。评审的设备千差万别，**对比度是最容易被忽视但最影响体验的细节**。

---

> **12 条踩坑小结**：从构建到样式、从数据到动画、从 PC 到移动，一个完整的前端项目会触碰到浏览器和框架的各种边界。记录这些坑，不仅是为了"展示做了什么"，更是证明——**这个作品真的被打磨过**。

## 🧠 设计中的关键决策（也是评审关注点）

> 每个决策都说明"为什么选 A 而不是 B"，体现产品和技术上的权衡思考。

---

### 决策 1：**纯前端方案 vs. 加后端**

| 维度 | 纯前端 + localStorage | 后端 + 数据库 |
|------|----------------------|---------------|
| 部署成本 | 双击 HTML 即可运行 | 需要服务器、域名、数据库配置 |
| 演示速度 | 0 秒上手 | 评委需登录/注册/等待接口响应 |
| 数据持久化 | localStorage，单设备可用 | 云端，多设备同步 |
| 开发复杂度 | 低（一个人可以搞定） | 中（前后端 + 部署 + 运维） |
| 可扩展性 | 有上限（受 localStorage 5MB 限制） | 无限扩展 |

**🔹 最终选择：纯前端**

**理由**：比赛评审最看重"能否快速看到完整效果"。一个需要注册登录、需要服务器运行的作品，评委的耐心只有 10 秒。纯前端方案打开即玩，**把评审的"认知负荷"降到最低**。

**妥协**：排行榜用虚拟用户数据（做真实的排行榜需要登录 + 后端同步，投入产出比低）。如果产品真的上线，迁移到 Supabase / Firebase 的成本很低。

---

### 决策 2：**接入第三方 AI API vs. 只用规则引擎**

| 维度 | 第三方 AI API | 纯规则引擎 |
|------|---------------|-----------|
| 回答质量 | 自然、有温度、可处理任意问题 | 生硬、关键词匹配、容易被"识破" |
| 门槛 | 需要 API Key / 网络环境 | 零门槛，离线可用 |
| 成本 | 免费额度基本够用，但 Key 管理是问题 | 零成本 |
| 展示效果 | "真的有 AI"，评审印象加分 | 像"高级版搜索框" |
| 稳定性 | 依赖网络 + 第三方服务 | 100% 可控 |

**🔹 最终选择：双轨架构（两者都要）**

**理由**：真实 API 让作品"看起来像 AI 产品"，满足评审对"AI 能力"的预期；规则引擎保证"无 Key、无网络也能演示"。两者不是互相替代的关系，而是**互补**。

**架构**：
- 用户配置了 API Key → 走真实 AI 对话（硅基流动 / DeepSeek / 智谱 等 4 大提供商可选）
- 用户未配置 API Key → 走知识库问答 + 推荐引擎（`knowledgeQA.ts` 中的规则匹配）
- 两套模式共享同一 UI：消息气泡、打字指示、上下文记忆

---

### 决策 3：**Recharts 做可视化 vs. 手写 SVG / D3 / Chart.js**

| 方案 | 上手成本 | React 集成 | 可定制性 | 包体积 |
|------|---------|-----------|---------|--------|
| **Recharts** | 低（组件化 API） | 原生 React 组件 | 中（可配 props） | 约 300KB gzip |
| 手写 SVG | 高（需要数学计算） | 需要自己封装 | 极高 | 0（但有开发时间） |
| D3.js | 高（学习曲线陡） | 需要桥接层 | 极高 | 约 250KB gzip |
| Chart.js | 中 | 需要 react-chartjs-2 桥接 | 中 | 约 60KB gzip |

**🔹 最终选择：Recharts**

**理由**：
1. **组件化 API** 和 React 的开发心智完全一致——`<RadarChart><PolarAngleAxis /><Radar dataKey="..." /></RadarChart>`，和写 `<div><button>Click</button></div>` 没区别
2. **饼图 + 雷达图 + 折线图**三个组件正好覆盖本产品的可视化需求（公益类型分布/六维画像/本周时长趋势），没有"杀鸡用牛刀"
3. **文档完善**，官方示例代码复制粘贴即可跑通，对 3 天开发周期的项目来说是关键

**妥协**：失去了完全自定义动画和形状的自由度。但对公益打卡产品来说，**"能用、好看、快速"比"技术上最酷"更重要**。

---

### 决策 4：**游戏化设计 vs. 纯工具化**

| 元素 | 无游戏化 | 有游戏化（当前方案） |
|------|---------|-------------------|
| 用户动机 | "我应该做公益"（外部压力） | "我想打卡解锁下一个徽章"（内在动机） |
| 次日留存 | 低（用完即走） | 中（连续打卡机制 + 成就收集欲） |
| 数据可视化 | 纯统计数字 | 等级进度条 + 徽章墙 + 排行榜 |
| 开发复杂度 | 简单 | 需要设计规则 + 成就检测逻辑 |

**🔹 最终选择：游戏化（挑战 + 成就 + 等级 + 排行榜）**

**理由**：公益行为的核心痛点不是"不知道怎么做"，而是**"做了也没反馈，很难坚持"**。游戏化机制就是为了解决这个问题：

- **7 日挑战**：培养习惯的黄金周期是 7 天，设计"连续打卡 7 天解锁成就"
- **18 个成就徽章**：覆盖首次行动、累计时长、分类任务、连续打卡、特殊成就 5 大类
- **等级成长**：Lv.1 新手 → Lv.6 公益大师，每级需要累计时长递增，有明确目标
- **排行榜**：和虚拟用户对比（因为纯前端无法做真实多人排名），提供"比昨天的自己更好"的参照

这不是"为了好玩而加的装饰"，而是**产品核心功能的一部分**——让坚持变得有趣。

---

### 决策 5：**单文件打包 vs. 多文件部署**

| 方案 | 产物 | 部署方式 | 评审体验 | 适用场景 |
|------|------|---------|---------|---------|
| **单文件** | 1 个 `index.html`（~1MB） | 双击 / 拖进浏览器 | ⭐⭐⭐⭐⭐ 零门槛 | 比赛作品、演示 Demo |
| 多文件 | `index.html` + `assets/*.js` + `assets/*.css` | 需要 HTTP 服务器 | ⭐⭐ 有配置门槛 | 生产环境、正式上线 |

**🔹 最终选择：单文件打包（vite-plugin-singlefile）**

**理由**：比赛评审的操作路径通常是——收到一个压缩包 → 解压 → 找到 index.html → 双击打开。如果是多文件，他们需要：
1. 安装 Node.js（如果没有）
2. 运行 `npm install && npm run preview`
3. 打开 `http://localhost:4173`

这三步中的任何一步都可能让评审"算了，下一个"。**单文件方案把评审的操作路径压缩到"解压 → 双击"两个动作**。

**技术实现**：
- `base: './'` → 所有资源路径改为相对路径
- `assetsInlineLimit: 100 * 1024` → 100KB 以内的资源全部内联为 base64
- `viteSingleFile()` → JS/CSS 注入到 HTML 的 `<script>` / `<style>` 标签
- `sourcemap: false` → 不生成 .map 文件，进一步减小体积

**妥协**：失去了 HTTP 缓存带来的二次访问速度提升（但对一次性评审场景无影响）；失去了代码分割的优化（但 1MB 在现代网络下完全可接受）。

---

> **关键决策小结**：这 5 个决策贯穿项目全程——从技术选型到产品定位。它们的共同点是：**在"理想方案"和"现实约束"之间选择了后者**。做产品不是追求技术上的最优解，而是追求**场景下的最佳实践**。

VolunteerMate 的开发过程展示了**一个完整的前端项目从设计到落地的全流程**：从选题分析、技术选型、架构设计、组件开发、AI 能力集成、交互打磨，到最终解决构建和部署问题。

4. 技术选型不必追求"最新最强"，**合适 + 可演示**才是硬道理

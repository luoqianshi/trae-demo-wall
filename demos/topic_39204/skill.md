# 阶段化 MVP 交付技能

## 目标

将需求拆成多个可独立验收的功能阶段，并严格执行以下规则：

1. 先验证需求文档可行性，再开始编码。
2. 每个功能都先交付可运行的 MVP，不追求一步到位。
3. 每完成一个功能阶段，必须暂停，等待人工验证。
4. 人工验证通过后，再生成合规提交日志并提交代码。
5. 未经人工确认，不得自动进入下一个功能阶段。
6. 每完成一个阶段，都要同步更新 `项目说明书-说明书.md`。

## 仓库与分支规则

仓库地址固定为 `https://gitee.com/lsm_first/qn.git`。

分支职责固定如下：

- `main`：只保留人工验收通过、可对外演示、可回滚的稳定代码。
- `develop`：当前功能阶段的集成分支。一个阶段完成并通过验证后，再合入 `main`。
- `feat/<feature-name>`：单个功能的主开发分支。一次只做一个功能 MVP。
- `feat/<feature-name>-sub<index>` 或 `fix/<feature-name>-<topic>`：当前功能下的子功能或修复分支。只允许从当前功能分支切出，完成后先回合到当前功能分支。

执行流必须接近下述结构：

1. `main` 创建项目主线。
2. 从 `develop` 切出当前功能分支 `feat/<feature-name>`。
3. 若当前功能还需拆子任务，从 `feat/<feature-name>` 再切 `feat/<feature-name>-sub1`、`feat/<feature-name>-sub2` 或 `fix/...`。
4. 子分支先合回当前功能分支。
5. 当前功能分支完成自测后，合回 `develop`。
6. 进行一次独立验收。
7. 人工确认通过后，补齐说明书、提交日志，再把该阶段从 `develop` 合入 `main`。
8. 未收到“可以继续下一个功能”的明确指令前，停止自动推进。

## 必做顺序

每次接到项目型需求，都按下面顺序执行：

1. 检查目录是否已初始化为 git 仓库；如没有，先初始化并绑定远程。
2. 检查 `前端.md`、`后端.md` 是否可执行；先修正文档中明显过度设计、环境依赖过重、版本表述不稳定的问题。
3. 创建或更新 `项目说明书-说明书.md`，先写：
   - 项目目标
   - 当前可行性结论
   - 阶段划分
   - 阶段记录模板
4. 先定义“功能 1”的 MVP 范围。
5. 先按前端需求补指定后端，再按后端接口补指定前端。
6. 完成后执行本地验证。
7. 输出人工验收说明，等待确认。
8. 只有在人工确认通过后，才生成合规提交内容并实际提交。
9. 提交后更新说明书中的阶段记录。
10. 停止，等待用户允许进入下一个功能。

## 文档可行性校正规则

看到前后端方案时，优先做可行性校正，不要直接照抄执行。

校正原则：

- 把“全功能大而全”改成“每功能先 MVP 跑通”。
- 把依赖重、模型大、设备要求高的方案改成可替换结构，先保留接口和 mock/fallback。
- 把“最新版本”“最新模型”改成“以仓库初始化时可安装稳定版本为准”。
- 把强依赖外网、GPU、系统环回音频的能力放到增强阶段，不放在首个 MVP 阶段作为阻塞项。
- 前后端都要先保留最小可演示链路：状态流转、消息协议、基础界面、基础接口、错误处理。

如果文档与现实环境冲突，优先保留：

1. 能运行
2. 能验证
3. 能提交
4. 能扩展

## MVP 拆分规则

每个功能阶段都必须满足以下最小标准：

- 有明确输入
- 有明确输出
- 有可运行路径
- 有本地验证方式
- 有失败回退方案

优先拆分方式：

1. 协议先行：先定义接口和消息结构。
2. 骨架先行：先搭主进程、服务入口、基础页面、基础状态管理。
3. 假数据先行：外部依赖未就绪时，先用 mock 数据跑通链路。
4. 单功能闭环：一个阶段只完成一个用户可感知能力。

禁止事项：

- 未验证当前阶段就连续做下一个阶段
- 在一个 commit 中混入多个不相关功能
- 跳过说明书更新
- 跳过人工验收
- 直接把未经验证的 `develop` 内容合入 `main`

## 说明书更新规则

`项目说明书-说明书.md` 是主线文档，每阶段都要追加记录，不覆盖历史。

每完成一个阶段，必须追加：

1. 阶段名称与日期
2. 该阶段的目标
3. 根据前端编写的指定后端内容
4. 根据后端编写的指定前端内容
5. 本地验证结果
6. 人工验收结论
7. 对应分支名、commit id、合规提交日志

## 验证与暂停规则

每阶段完成后必须先做本地验证，至少包含：

- 启动验证
- 核心流程验证
- 错误分支验证
- 文档同步检查

然后向用户输出：

- 本阶段完成了什么
- 如何人工验证
- 当前风险或未完成项
- 建议的提交标题与提交说明

输出后立即暂停，不得自动继续下一阶段。

## 合规提交规则

提交必须满足持续交付和单一职责要求。

commit 要求：

- 一次 commit 只对应一个功能阶段或一个明确修复。
- 使用清晰前缀：`feat:` `fix:` `docs:` `refactor:` `test:`
- message 必须说明本次变更的功能边界。

推荐格式：

```text
feat: 完成功能1 MVP 字幕链路
fix: 修复功能1 WebSocket 重连状态不同步
docs: 更新阶段1说明书与验收记录
```

PR 或合并说明必须包含：

```markdown
## 标题
一句话说明本阶段交付内容

## 功能描述
本阶段新增或修复了什么

## 实现思路
本阶段如何落地，哪些能力先用 MVP 或 mock 实现

## 测试方式
如何在本地复现和验证
```

如果当前环境没有远程 PR 条件，先在本地生成同等结构的“合规提交日志”，并写入说明书。

## 推荐阶段模板

对于“前端 + 后端”项目，优先按以下阶段推进：

1. 阶段 0：文档校正、仓库初始化、说明书建立
2. 阶段 1：后端最小服务入口 + 前端最小壳层 + WebSocket 协议打通
3. 阶段 2：翻译消息流 MVP + 字幕展示 MVP
4. 阶段 3：纠错消息流 MVP + 高亮展示 MVP
5. 阶段 4：控制面板配置 MVP + 后端状态控制 MVP
6. 阶段 5：真实音频/ASR/TTS 能力逐项替换 mock
7. 阶段 6：稳定性、验收、README、演示材料补齐

## 输出风格

对用户汇报时保持简洁、直接、可执行。

优先输出：

1. 当前阶段
2. 做了什么
3. 如何验证
4. 是否可以提交
5. 是否允许进入下一阶段

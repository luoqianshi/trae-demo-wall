# 项目说明书

## 1. 项目概述

项目名称：AI 同声传译助手

目标：构建一个包含桌面前端与实时翻译后端的单仓库项目，采用“逐功能 MVP、人工验收、合规提交”的方式推进，确保每一阶段都可运行、可验证、可提交。

仓库地址：`https://gitee.com/lsm_first/qn.git`

当前仓库状态：

- 本地仓库已初始化
- 远程 `origin` 已绑定到 `https://gitee.com/lsm_first/qn.git`
- 当前目录仅包含规划文档，尚未开始功能代码开发
- 当前尚无首个提交，`develop` 分支将在阶段 0 首次合规提交后从 `main` 切出
- 后续开发需严格遵守 `main` / `develop` / `feat/*` 分支流程

## 2. 文档可行性结论

### 2.1 总体判断

前端方案和后端方案都具备明确的目标与模块划分，但当前版本存在明显的“直接按最终态设计”的倾向，不适合作为首轮开发计划原样执行。必须先收敛为可运行的 MVP 链路，再逐阶段替换真实能力。

### 2.2 前端文档可行性评估

可保留内容：

- Electron 透明字幕层方向合理
- React 组件化结构合理
- WebSocket 实时通信方向合理
- 控制面板、字幕层、纠错高亮、设置持久化拆分合理

需要调整内容：

- “最新稳定版”表述不适合作为固定执行条件，实际应以初始化时可安装版本为准
- 鼠标穿透、透明窗口、置顶等桌面能力依赖操作系统差异，不应作为首阶段阻塞项
- 首阶段不应同时完成完整设置、样式、TTS 开关、所有动画细节
- 首阶段应先完成最小窗口骨架、字幕展示容器、WebSocket 消息接收与状态显示

前端调整结论：

- 阶段 1 只做 Electron 壳层、React 页面骨架、基础字幕区域、基础连接状态展示
- 阶段 2 再做流式字幕与最近多句保留
- 阶段 3 再做纠错高亮
- 阶段 4 再做控制面板设置项与持久化

### 2.3 后端文档可行性评估

可保留内容：

- FastAPI + WebSocket 结构合理
- 管道式拆分为音频、ASR、翻译、TTS、状态管理是可扩展方案
- 明确消息协议是正确方向

需要调整内容：

- `faster-whisper`、`Silero VAD`、`IndexTTS2`、系统音频环回都属于较重依赖，不适合作为首阶段前置条件
- DeepSeek 实时翻译依赖外部 API，不应阻塞首个 MVP
- “端到端 3~5 秒”属于性能目标，不应放在首个阶段作为验收门槛
- 首阶段应先使用 mock 文本流和 mock 状态消息打通端到端链路

后端调整结论：

- 阶段 1 只做 FastAPI 服务入口、WebSocket 协议、基础状态切换、mock 消息推送
- 阶段 2 再接入翻译消息流的真实接口适配层，但允许先保留 mock/fallback
- 阶段 3 再增加纠错消息结构与上下文管理
- 阶段 4 再增加控制指令处理
- 阶段 5 再逐项替换真实音频、ASR、TTS 能力

### 2.4 联合执行结论

本项目可行，但必须采用以下执行原则：

1. 先后端、后前端，但两者在同一阶段内形成闭环
2. 每个阶段只完成一个用户可感知能力
3. 每个阶段先本地验证，再人工验收，再合规提交
4. 未收到人工确认，不进入下一个阶段

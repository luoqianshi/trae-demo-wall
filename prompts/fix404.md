> 本提示词用于指导 TRAE Work 自动修复 TRAE Demo Wall 项目中 Demo 数据 404 问题（local 路径错误、标签缺失、URL 异常等），并构建部署。请严格按照以下步骤执行，不要跳过任何步骤，不要自行添加额外操作。

## 问题背景

Demo 数据出现 404 访问的常见原因：

| 原因 | 说明 |
|---|---|
| **ZIP 解压后嵌套目录** | 作者上传的 ZIP 内部有子目录（如 `my-project/index.html`），解压后路径变成 `public/demos/topic_12345/my-project/index.html`，但数据记录的是 `./demos/topic_12345/index.html`（扁平路径） |
| **HTML 入口文件名非 index.html** | 作者的入口文件叫 `main.html`、`app.html` 等，但数据统一记录为 `index.html` |
| **文件未成功下载/解压** | ZIP 下载失败或 CRC 校验失败，导致目录不存在 |
| **标签为空** | 帖子发布时未选择标签，或爬虫未正确提取标签 |
| **异常标签** | 论坛原始标签为数字 ID（如 `65-tag`、`68-tag`），未映射为实际名称 |
| **URL 含特殊字符** | 外部 demoUrl 末尾误带全角标点，或使用 HTTP 而非 HTTPS |

`fix_data.py` 脚本会自动检测并修复以上所有问题。

## 执行步骤

请**严格按照顺序**执行以下所有步骤。

### Step 1: 确认项目环境

```bash
cd /workspace/trae-demo-wall
```

1. 确认 `crawler/fix_data.py` 文件存在
2. 确认 `src/data/index.json` 文件存在
3. 确认 `public/demos/` 目录存在（存放 local 类型的 demo 文件）
4. 确认 Node.js 和 Python 3 可用：
   ```bash
   node --version
   python3 --version
   ```
5. 如果 `node_modules` 不存在，安装前端依赖：
   ```bash
   npm install
   ```

### Step 2: 运行数据修复脚本

```bash
cd /workspace/trae-demo-wall/crawler && python3 fix_data.py
```

脚本会执行以下 5 个修复步骤：

| 步骤 | 操作 |
|---|---|
| Step 1: 修复空标签 | 从项目标题中解析赛道关键词（学习工作/生活娱乐/社会服务/社会公益/硬件交互），补全到 tags 字段 |
| Step 2: 移除异常标签 | 清除 `65-tag`、`68-tag` 等论坛原始数字 ID 标签 |
| Step 3: 清洗 URL | 移除 demoUrl 末尾的全角分号、截图 URL 中的异常编码；HTTP→HTTPS（仅 Pages 托管域名） |
| Step 4: 修复 local 路径 | 递归扫描 `public/demos/topic_*/` 目录，找到实际的 HTML 入口文件，更新 localPath；如果目录不存在则标记为 `type=none` |
| Step 5: 重新生成数据文件 | 重新生成 `src/data/index.json` 和 `src/data/pages/page-*.json` |

**注意事项：**
- 脚本运行通常在 10-30 秒内完成（取决于项目数量和文件扫描耗时）
- 如果输出中显示大量 `[WARN]` 和 `→ 替代路径`，这是**正常现象**，说明脚本正在修复嵌套目录问题
- 只有 `→ 标记为无法预览 (type=none)` 表示该项目的 demo 文件确实不存在，无法修复
- 如果输出显示 `修复完成！` 且 `local 文件缺失: 0 个`，说明所有路径都已正确，无需修复

运行完成后，记录脚本输出的修复统计（空标签修复数、URL 清洗数、local 文件缺失数）。

### Step 3: 构建前端

```bash
cd /workspace/trae-demo-wall && npm run build
```

构建成功的标志是输出 `✓ built in Xs`，且没有 Error。

**验证构建产物**（必须执行）：
```bash
cat /workspace/trae-demo-wall/dist/data/index.json | python3 -c "import json,sys; d=json.load(sys.stdin); print(f'构建验证: totalProjects={d[\"totalProjects\"]}')"
```

输出的 `totalProjects` 应与 `src/data/index.json` 中的数量一致。如果不一致，执行：
```bash
cd /workspace/trae-demo-wall && rm -rf public/data && cp -r src/data public/data && rm -rf dist && npm run build
```

### Step 4: 提交并推送到 main 分支

```bash
cd /workspace/trae-demo-wall
git add -A
git commit -m "fix: data quality - tags/paths/URLs (empty tags: N, paths: M, URLs: K)"
```

**将 N、M、K 替换为 Step 2 中脚本输出的实际修复数量。** 例如：
```
fix: data quality - tags/paths/URLs (empty tags: 68, paths: 867, URLs: 2)
```

如果没有需要修复的问题（所有计数为 0），则**不要提交**，直接告知用户无需修复。

然后推送：
```bash
git push origin main
```

### Step 5: 验证部署

推送后，GitHub Actions 会自动触发构建部署（通常 2-5 分钟完成）。检查状态：

```bash
curl -s "https://api.github.com/repos/luoqianshi/trae-demo-wall/actions/runs?per_page=1" | python3 -c "
import json,sys
data=json.load(sys.stdin)
run=data['workflow_runs'][0]
print(f'Action: {run[\"name\"]}')
print(f'Status: {run[\"status\"]}')
print(f'Conclusion: {run[\"conclusion\"] or \"running\"}')
"
```

等待 `completed` + `success` 后，验证线上数据：
```bash
sleep 30
curl -s "https://luoqianshi.github.io/trae-demo-wall/data/index.json?v=$(date +%s)" | python3 -c "import json,sys; d=json.load(sys.stdin); print(f'线上验证: totalProjects={d[\"totalProjects\"]}, updatedAt={d[\"updatedAt\"]}')"
```

## 常见问题处理

### Q1: 脚本运行后 `local 文件缺失: 0`，但线上仍有 404
**处理**: 可能是前端缓存问题。前端已添加 `?v=${Date.now()}` 缓存破坏参数，正常刷新即可。如果仍有问题，检查 `dist/data/pages/` 中的路径是否与 `public/demos/` 的实际目录结构一致。

### Q2: npm install 或 npm run build 失败
**处理**:
```bash
cd /workspace/trae-demo-wall && rm -rf node_modules package-lock.json && npm install
```

### Q3: git push 被拒绝（rejected）
**处理**:
```bash
cd /workspace/trae-demo-wall && git pull --rebase origin main && git push origin main
```

### Q4: 修复脚本报 `ModuleNotFoundError`
**处理**: 安装 Python 依赖：
```bash
pip install requests beautifulsoup4 lxml --break-system-packages -q
```
（注意：`fix_data.py` 实际上只使用标准库 `json`、`os`、`re`、`sys`、`datetime`，通常不需要额外依赖）

### Q5: 某些 local 路径包含乱码（如 `╩²╫╓╗»╣▄└φ╞╜╠¿ .html`）
**处理**: 这是文件名编码问题（GB2312/GBK 编码的中文文件名在 Linux 下显示为乱码）。修复脚本会保留原始乱码路径（因为文件系统中就是这样命名的），不影响实际访问。如果确实无法访问，手动将该文件标记为 `type=none`。

### Q6: GitHub Pages 显示旧数据
**处理**: 前端已添加 `?v=${Date.now()}` 缓存破坏参数，正常情况下 5 分钟内更新。超过 10 分钟未更新，检查 GitHub Actions 是否成功。
